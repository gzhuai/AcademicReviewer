# 学术竞赛文稿审查智能体 —— 完整设计方案 v1.3

---

## 一、核心定位与设计原则

### 1.1 核心定位

这个系统的本质是一个 **"AI 学术导师 + 质量门禁"**，而非代写工具。它的核心价值在于：

> 在不替代学生思考的前提下，系统性地发现文稿中的问题，并以**可操作的反馈**指导学生自主改进。

### 1.2 设计原则（六项铁律）

| 原则 | 说明 |
|------|------|
| **学生主权** | AI 只做批注和建议，绝不直接改写正文，确保作品仍为学生原创 |
| **分层审查** | 从宏观（论点结构）到微观（语法拼写），逐层把关，避免挂一漏万 |
| **竞赛对齐** | 每次审查都与具体竞赛的评分标准（rubric）严格对标 |
| **可解释性** | 每个建议必须附带理由和评分维度，让学生"知其所以然" |
| **迭代闭环** | 支持多轮提交→审查→修改→再审，跟踪质量收敛趋势 |
| **数据驱动校准** | 用获奖/失败样本持续反推评委隐性偏好，校准配置文件（v1.3 新增） |

## 二、系统架构：双管线协作体系

### 架构全景图

v1.3 的核心架构演进：将系统分为两条独立的管线——**审查管线**（在线，实时审查学生文稿）和**校准管线**（离线，赛季后批量分析样本、更新配置）。两条管线通过**配置文件层**解耦。

```
┌─────────────────────────────────────────────────────────────────┐
│                        AcademicReviewer                         │
├─────────────────────────────────────────────────────────────────┤
│                                                                 │
│  ┌──────────────────────┐          ┌──────────────────────┐    │
│  │   审查管线（在线）      │          │   校准管线（离线）      │    │
│  │                      │          │                      │    │
│  │  学生文稿 ─→ Orchestrator ─→   │   获奖/失败文章集 ─→     │    │
│  │                │       │       │   Calibration Engine   │    │
│  │    ┌───┬───┬───┼───┬───┤       │         │              │    │
│  │    ▼   ▼   ▼   ▼   ▼   ▼       │    ┌─────▼─────┐       │    │
│  │   A1  A2  A3  A4  A5  A6  A7   │    │ 特征提取   │       │    │
│  │    │   │   │   │   │   │   │    │    │ 差异分析   │       │    │
│  │    └───┴───┴───┴───┴───┴───┘    │    │ 规则生成   │       │    │
│  │             │                  │    │ Diff 输出  │       │    │
│  │             ▼                  │    └─────┬─────┘       │    │
│  │       审查报告                  │          │              │    │
│  └──────────────┬───────────────┘          │              │    │
│                 │                          ▼              │    │
│                 │              ┌──────────────────┐       │    │
│                 │   读取配置    │    配置文件层      │ ◀写入  │    │
│                 └─────────────▶│  · rubric JSON    │───────┘    │
│                               │  · structure schema│          │
│                               │  · evidence patterns│         │
│                               │  · style guide     │          │
│                               │  · citation rules  │          │
│                               └──────────────────┘          │
│                                                                 │
└─────────────────────────────────────────────────────────────────┘
```

**关键解耦**：配置文件是两条管线之间的唯一数据契约。审查管线不关心配置是怎么来的；校准管线不关心配置被谁消费。

### 2.1 审查管线：七个专业 Agent

采用 **编排器 + 专家智能体池（Orchestrator + Agent Pool）** 架构。每个 Agent 从配置文件层读取对应配置，执行单篇文稿的专项审查。

```
┌──────────────────────────────────────────────────────────┐
│                     Orchestrator                         │
│              (任务调度 / 结果聚合 / 质量报告)               │
└──────┬───────┬───────┬───────┬───────┬───────┬──────────┘
       │       │       │       │       │       │
   ┌───▼──┐┌──▼───┐┌──▼───┐┌──▼───┐┌──▼───┐┌──▼──────┐
   │Rubric││Logic ││Argue ││Style ││Citation││Original│
   │Parser││&Flow ││&Evid ││&Voice││Checker││  Check  │
   └──────┘└──────┘└──────┘└──────┘└───────┘└─────────┘
```

#### Agent 1: Rubric Parser（评分标准解析器）
- **输入**：竞赛官方 rubric（PDF/图片/文本）
- **输出**：结构化的评分维度树 + 权重 + 关键词
- **能力**：将模糊的评分描述（如 "demonstrates critical thinking"）转化为可检查的具体指标
- **配置来源**：`rubric_calibrated.json`（由 Calibration Engine 生成）

#### Agent 2: Structure & Logic（结构与逻辑审查）
- 检查论文整体骨架：引言→文献综述→方法→论证→结论的完整性
- 检测段落间的逻辑衔接（是否存在跳跃、重复、缺失环节）
- 评估 thesis statement 的清晰度和全文一致性
- **配置来源**：`structure_schema_calibrated.json`

#### Agent 3: Argument & Evidence（论点与证据审查）
- 核查每个 claim 是否有充分 evidence 支撑
- 识别逻辑谬误（因果倒置、稻草人、以偏概全等）
- 评估证据来源的权威性和时效性
- 依据校准后的 fatal_defects 标记「有则必死」的硬伤
- **配置来源**：`evidence_patterns.json`（按赛事类型）

#### Agent 4: Writing Style & Voice（写作风格与语气）
- 检查写作风格规范性（避免口语化、情绪化表达）
- 术语使用的一致性
- 被动语态/主动语态的合理分布
- 是否符合目标竞赛的预期风格：
  - 学术-技术型（如 ISEF）：干练精确，被动语态可接受
  - 学术-思辨型（如 John Locke）：优雅有力，展示哲学思辨魅力
  - 商务-金融型（如 WGHS）：简洁果断，主动语态优先，数据驱动叙事
- **配置来源**：`style_guide.json`（按赛事类型）

#### Agent 5: Citation & Format（引用与格式审查）
- 检查所有引用是否完整（有 cite 必有 reference，反之亦然）
- 验证引用格式（APA / MLA / Chicago 等）的一致性
- 检测可疑的 AI 生成引用（虚假文献检测）
- **配置来源**：`citation_rules.json`

#### Agent 6: Originality Guardian（原创性守护）
- 检测潜在抄袭/过度借鉴段落（与内部历史稿件库比对）
- 识别大段 AI 生成痕迹（语气突变、模板化结构）
- 确保学生个人 voice 得以保留
- **配置**：无需配置（L0 通用引擎）

#### Agent 7: Language Polishing（语言精修）—— 唯一允许改写的 Agent
- 语法、拼写、标点修正
- 中式英语 → 地道表达的建议
- 长难句拆分建议
- **严格以批注模式工作，不直接改动原文**
- **配置**：仅 L1 风格开关（学术模式 / 商务模式）

### 2.2 配置层：Agent 与配置文件的映射

配置文件是审查管线和校准管线的数据契约。它在 v1.3 中的角色从「静态模板」升级为「持续校准的活文档」。

| 配置文件 | 消费方 | 生成/更新方 | 校准频率 |
|----------|--------|------------|----------|
| `rubric_calibrated.json` | Agent 1 | Calibration Engine | 每赛季校准 |
| `structure_schema_calibrated.json` | Agent 2 | Calibration Engine | 每赛季校准 |
| `evidence_patterns.json` | Agent 3 | Calibration Engine | 每次有新样本 |
| `style_guide.json` | Agent 4 | Calibration Engine | 每赛季或按需 |
| `citation_rules.json` | Agent 5 | 手动定义 | 无需校准 |
| 无（通用引擎） | Agent 6, 7 | 无 | 无需配置 |

```
配置文件示例（evidence_patterns.json，科研型）:

{
  "type": "research",
  "season": "2026",
  "evidence_standards": {
    "quantitative": {
      "expect_p_value": true,
      "p_value_threshold": 0.05,
      "expect_effect_size": true,            // ← 样本反推：获奖者 d=0.91
      "expect_sensitivity_analysis": true     // ← 样本反推：获奖者 d=0.85
    },
    "source_quality": {
      "min_primary_sources": 12              // ← 样本反推：获奖者均值 18
    }
  },
  "fatal_defects": [                         // ← 完全来自失败样本
    { "defect": "no_control_group", "loser_rate": 35, "winner_rate": 2 },
    { "defect": "citation_appears_fabricated", "loser_rate": 18, "winner_rate": 1 }
  ],
  "confidence": 0.82,
  "based_on": "15_winners + 12_losers + 5_external_winners"
}
```

### 2.3 校准管线：Calibration Engine

#### 2.3.1 定位

Calibration Engine 是一个**纯统计管线**，不调用 LLM。它在赛季后以批量离线模式运行，从获奖/失败文章的差异中提取可量化的校准信号，输出更新后的配置文件。

#### 2.3.2 核心方法论：Cohen's d 差异分析

不凭感觉说「获奖文章好像更重视 X」，而是用效应量（effect size）量化：

```
d = (M_winners - M_losers) / SD_pooled

d > 0.8  → 大效应 — 决定性差异 → 写入配置文件 critical 项
d > 0.5  → 中效应 — 重要差异 → 写入 major 项
d > 0.2  → 小效应 — 轻微差异 → 写入 minor 项
d < 0.2  → 无差异 — 评委可能不关心 → 不写入
```

#### 2.3.3 四阶段处理流程

```
┌──────────────────────────────────────────────────────────┐
│              Calibration Engine 四阶段流程                 │
├──────────────────────────────────────────────────────────┤
│                                                          │
│  Step 1: 特征提取（全自动，离线）                          │
│  ├── 对获奖/失败/外部获奖文章逐篇结构化特征扫描               │
│  ├── 特征维度：句式参数、引用密度、段落骨架覆盖度、            │
│  │           论证密度、被动语态占比、词汇多样性、              │
│  │           逻辑标志词密度、过渡句频率、证据多样性...         │
│  └── 输出：特征矩阵（篇 × 特征）                            │
│                                                          │
│  Step 2: 分组差异分析                                     │
│  ├── 分组：我方获奖 / 我方失败 / 外部获奖                    │
│  ├── 对每个特征 × 每组对，计算 Cohen's d                    │
│  ├── 交叉验证：我方获奖 ∩ 外部获奖 = 确定性信号              │
│  │            外部获奖独有 = 我方辅导盲区                   │
│  └── 输出：效应量排名表 + 置信区间                          │
│                                                          │
│  Step 3: 规则生成                                         │
│  ├── d > 0.8 → 写入 critical 项                           │
│  ├── d > 0.5 → 写入 major 项                              │
│  ├── 失败文章独有而获奖文章几乎不出现 → fatal_defects       │
│  ├── 为每条规则标注 confidence + data_source               │
│  └── 输出：新版本配置文件（JSON）                           │
│                                                          │
│  Step 4: Diff 生成（面向运营人员）                          │
│  ├── 与上一赛季配置文件对比                                  │
│  ├── 标注每条变更的类型：新增 / 参数调整 / 权重修正            │
│  ├── 附带变更理由（效应量 + 数据来源）                       │
│  └── 输出：human-friendly 校准建议报告                      │
│                                                          │
│  备注：全程不调用 LLM。这是统计 + 规则，不是 AI。            │
└──────────────────────────────────────────────────────────┘
```

#### 2.3.4 三类校准信号

| 信号类型 | 来源 | 示例 | 写入配置文件位置 |
|----------|------|------|-----------------|
| **正向信号** | 获奖文章共同特征 | "获奖者 Methods 段都有可复现性声明 (d=0.78)" | `quality_expectations` |
| **负向信号** | 失败文章独有缺陷 | "失败者 35% 缺少控制组，获奖者仅 2%" | `fatal_defects` |
| **交叉验证信号** | 我方获奖 ∩ 外部获奖 | "gap statement 使用率双方均 > 80%" → 高置信 | `confidence: high` |

### 2.4 各 Agent 校准需求矩阵

并非所有 Agent 都需要校准。校准深度由「评委对该维度是否存在隐性偏好」决定。

| Agent | 需要校准？ | 校准深度 | 校准方式 | 校准对象 | 校准频率 |
|-------|:---:|----------|----------|----------|----------|
| **A1 Rubric** | ✅ | 深 | 维度权重反推、子维度发现 | `rubric_calibrated.json` | 每赛季 1 次 |
| **A2 Structure** | ✅ | 中浅 | 二元检查项升级为「有+质量预期」 | `structure_schema_calibrated.json` | 每赛季 1 次 |
| **A3 Argument** | ✅✅ 强需 | 最深 | 论证模式反推、fatal_defects 挖掘 | `evidence_patterns.json` | 有新样本即触发 |
| **A4 Style** | ✅ | 中 | 风格特征区间统计 | `style_guide.json` | 每赛季或按需 |
| **A5 Citation** | ❌ | — | 规则驱动（APA/MLA等），无隐性偏好 | — | — |
| **A6 Originality** | ❌ | — | L0 通用引擎 | — | — |
| **A7 Language** | ❌ | — | L0 通用，仅风格开关为 L1 | — | — |

**解读**：

- **A3 是校准的重中之重**。不同赛事类型对「好论证」的定义有隐性标准（如 ISEF 获奖者用 p-value + effect size 双重报告，而官方 rubric 只写 "demonstrates critical thinking"），只有样本能揭示。且它不是赛季固定更新的节奏——每次有新样本入库都应触发微调。
- **A1 校准的是「评分标准」**（评委声称看重什么），**A3 校准的是「评分实质」**（评委实际看重什么）。两者缺一不可。
- **A2/A4 可放在第二阶段做**，不影响 MVP。先用启发式规则运行，有足够样本后再做定量校准。
- **A5 格式规则不需要校准**（APA 就是 APA），但引用数量/来源多样性偏好属于 A3 范畴。

### 2.5 竞赛类型五分类

系统将常见国际竞赛归为五大类型，每种类型驱动不同的 L2 审查逻辑（Agent 3 + Agent 4）：

| 类型 | Agent 3 审查重点 | Agent 4 风格标准 | 典型竞赛 |
|------|-----------------|-----------------|----------|
| **科研型** | 实验设计、数据显著性、可复现性、变量控制 | 学术-技术：干练精确、被动语态可接受 | ISEF, Regeneron STS, ISSDC |
| **思辨型** | 逻辑链完整性、哲学引用深度、反驳力度 | 学术-思辨：优雅有力、展示个人 voice | John Locke, Marshall Society |
| **社科型** | 一手调研方法论、案例分析深度、跨文化视角 | 学术-社科：调研叙事、平衡客观与洞察 | CTB, YOC, YIC |
| **历史型** | 一手史料引用、史学视角、contextualization | 学术-史学：史实精准、时代语境还原 | NHD, The Concord Review |
| **金融型** | 估值模型合理性、财务数据溯源、风险分析框架、图表质量 | 商务-金融：简洁果断、数据驱动、主动语态 | SIC, WGHS, YIS |

**金融型竞赛的特殊性：**

金融投资报告（如 WGHS、SIC、YIS）与学术论文存在本质差异，是五大类型中复用度最低的一类：

| 维度 | 传统学术论文 | 金融投资报告 |
|------|-------------|-------------|
| **写作目的** | 论证学术观点 | 说服投资决策 |
| **核心骨架** | Intro → Lit Review → Method → Results → Discussion | Executive Summary → Investment Thesis → Industry Analysis → Valuation → Portfolio → Risk Analysis |
| **证据类型** | 学术引用、实验数据、一手史料 | 财务数据、估值模型（DCF/Comparables）、SEC 文件、Bloomberg 数据 |
| **写作风格** | 学术英语，被动语态可接受，hedging language | 商务英语，短句优先（15-20词），主动语态，明确判断 + 证据支撑 |
| **图表角色** | 辅助说明 | **核心叙事载体**（K线图、财务模型、敏感性分析表） |
| **引用体系** | APA/MLA/Chicago 正式引用 | 脚注式数据来源标注，精确但不拘泥于格式 |
| **关键风险** | 逻辑谬误、证据不足 | 估值假设不合理、数据来源缺失、风险对冲策略缺失 |

### 2.6 精简方案与取舍分析

本章探讨 7 Agent 架构是否可精简，以及精简后的成本、性能和质量权衡。

#### 2.6.1 Token 成本与延迟测算

以一篇 **5000 词**的典型竞赛论文为基准：

| Agent | 输入 Token | 输出 Token | 输入内容 |
|-------|-----------|-----------|----------|
| 1. Rubric Parser | 3K | 1K | 仅 rubric 文本（不传全文） |
| 2. Structure & Logic | 8K | 2K | 全文 + 结构 schema |
| 3. Argument & Evidence | 8K | 3K | 全文 + rubric 维度 |
| 4. Writing Style & Voice | 8K | 1.5K | 全文 |
| 5. Citation & Format | 8K | 1.5K | 全文 + ref 列表 |
| 6. Originality Guardian | 10K | 1K | 全文 + 比对片段 |
| 7. Language Polishing | 8K | 3K | 全文 |

| 汇总 | 数值 |
|------|------|
| 总输入 Token | ≈ 53K |
| 总输出 Token | ≈ 13K |
| **单次审查总消耗** | **≈ 66K tokens** |
| 按 GPT-4o 价格估算 | **≈ $0.26/篇（约 ¥1.90）** |
| 月均审查 100 篇 | ≈ ¥190 |
| 年化成本 | ≈ ¥2,280 |

**结论：Token 成本在日常运营量级下几乎可忽略。真正的优化价值不在于省钱，而在于减少端到端延迟、降低工程复杂度、提高单次响应的可靠性和可调试性。**

> **关于 Calibration Engine**：校准时不调用 LLM，纯统计计算，无额外 token 成本。

**延迟对比：**

```
7 Agent（分轮并行）:
Round 1: A1+A2+A3 → 最长 6s
Round 2: A4+A5+A7 → 最长 5s
Round 3: A6 → 6s
─────────────────
总计 ≈ 17s
```

#### 2.6.2 Agent 合并可行性矩阵

| Agent X | 与 Agent Y 合并 | 判定 | 理由 |
|---------|----------------|------|------|
| A1 Rubric | A2/A3/A4/A5/A6/A7 | ❌ 不可 | Rubric 解析是前置任务，本质是"配置层"而非"审查层" |
| A2 Structure | A3 Argument | ⚠️ 可但存疑 | 结构看"骨架有没有"，论证看"肉好不好"。合并后 prompt 臃肿 |
| A2 Structure | A4 Style | ❌ 不可 | 宏微观分离是设计根基 |
| A3 Argument | A4 Style | ❌ 不可 | 审逻辑和审表达混在一起会互相污染 |
| **A4 Style** | **A7 Language** | **✅ 可合并** | 风格和语言是连续谱。代价：A7 改写权限需严格约束 |
| **A5 Citation** | **A6 Originality** | **✅ 可合并** | 同属"学术诚信"范畴。代价：实现方式不同（规则 vs 语义） |

**可行合并点**：
- **合并点①**：Agent 4 + Agent 7 → 统管"语言表达层"
- **合并点②**：Agent 5 + Agent 6 → 统管"学术诚信层"

#### 2.6.3 方案 A：5 Agent（推荐）

```

└──────┘    └─────────────────────────┘    └──────────┘    └──────────┘

                                                          ┌────▼─────┐
                                                          │ Stage5:  │
                                                          │ Final QC │
                                                          │ & Export │
```

#### Stage 1: 材料摄入与预处理（Intake）

**输入材料清单（v1.3 更新）：**

| 类别 | 材料 | 必须/可选 | 说明 |
|------|------|-----------|------|
| **学生文稿** | 完整英文稿件（.docx / .tex / .md） | ✅ 必须 | 支持带格式文档 |
| **竞赛信息** | 竞赛名称、类别、提交截止日期 | ✅ 必须 | 用于加载对应配置文件 |
| **评分标准** | 官方 rubric / judging criteria | ✅ 必须 | 有官方原文最佳；无可使用系统推断版本 |
| **字数要求** | 最低/最高字数限制 | 🟡 建议 | |
| **格式要求** | 字体、行距、页边距等 | 🟡 建议 | |
| **往年范文** | 该竞赛往届获奖作品 | 🔵 可选 | 用于风格对标和质量基准 |

#### Stage 2: 多 Agent 并行审查

审查时，每个 Agent 读取当前锁定的配置文件，执行专项审查。配置文件由 Calibration Engine 在上一赛季末生成，赛季中不修改。

```
Orchestrator 调度顺序（7 Agent 设计）:

Round 1（宏观）- 并行执行:
  ├── Agent 1: Rubric Parser → 读取 rubric_calibrated.json → 生成检查清单
  ├── Agent 2: Structure & Logic → 读取 structure_schema_calibrated.json
  └── Agent 3: Argument & Evidence → 读取 evidence_patterns.json

Round 2（微观）- 基于 Round 1 结果执行:
  ├── Agent 4: Writing Style → 读取 style_guide.json
  ├── Agent 5: Citation Check → 读取 citation_rules.json
  └── Agent 7: Language Polish → 通用引擎

Round 3（综合）:
  └── Agent 6: Originality Guardian → 通用引擎
```

#### Stage 3-5

反馈报告生成、学生修改辅助、终审与导出——与 v1.2 保持一致，无变化。

### 3.3 操作手册：赛季校准的具体动作

#### 3.3.1 你需要做的（人的操作）

```
赛季前校准窗口：

  Step 1 ─ [A1] 上传或确认官方 rubric
              动作：把官方 PDF 或文本放入对应竞赛目录
              耗时：5 分钟（有官方文件）或 跳过（无官方文件时用推断版）

  Step 2 ─ 标记样本集
              动作：在系统中标注文章分类
              ├── my_winners/  ← 我方辅导获奖文章（路径列表）
              ├── my_losers/   ← 我方辅导失败文章（路径列表）
              └── external_winners/ ← 公开的官方获奖文章（路径列表）
              耗时：10-20 分钟（取决于文章数量）

  Step 3 ─ 触发校准
              动作：点击「运行校准」或调用 Calibration Engine
              耗时：1 秒（触发后系统自动完成，分钟级等待即可）

  Step 4 ─ 审核校准建议 ⭐ 核心动作
              动作：
              ├── 打开系统生成的「校准建议报告」
              ├── 逐条查看建议：
              │   ├── 新增维度（如 "sensitivity_analysis_expected"）
              │   ├── 权重调整（如 methodology_weight: 0.20 → 0.25）
              │   ├── 阈值变更（如 min_primary_sources: 10 → 12）
              │   └── fatal_defects 更新
              ├── 对每条建议：采纳 / 拒绝 / 调整数值
              └── 确认锁定
              耗时：15-30 分钟

  Step 5 ─ 锁定版本
              动作：确认「锁定配置文件 vX.Y 用于本赛事赛季」
              耗时：1 秒
```

#### 3.3.2 系统自动完成的（不用你管）

- 文章预处理（分段、分句、特征提取）
- 所有特征的 Cohen's d 计算
- 获奖/失败/外部获奖三组交叉分析
- 校准建议生成（diff 格式）
- 配置文件历史版本快照
- 赛季校准日志自动写入
- 校准参数年度漂移趋势图

#### 3.3.3 赛季中和赛季后

| 时期 | 你的动作 | 系统动作 |
|------|----------|----------|
| **赛季中** | 零操作（专心辅导学生） | 审查管线按锁定配置运行 |
| **赛季后 1 周** | 标注获奖/失败名单 | — |
| **赛季后 2 周** | 审核校准建议，锁定新配置 | 运行 Calibration Engine，生成建议 |

## 四、技术实现建议

### 4.1 推荐的 Prompt 工程策略

每个 Agent 使用**结构化 Prompt 模板**，至少包含：

```
[Role] 你是XXX领域的竞赛评委，拥有15年评审经验
[Context] 这是参加XXX竞赛的高中生论文，竞赛rubric如下...
[Task] 请逐段审查本文的结构逻辑，重点关注...
[Output Format] 请严格按以下JSON格式输出...
[Constraints] 不要重写学生的句子，只指出问题和改进方向
[Examples] 好的反馈 vs 差的反馈示例...
```

### 4.2 与现有 LLM 的适配

鉴于在中国大陆运营，建议考虑：
- **模型层抽象**：同时支持 OpenAI / Claude / 国产模型（DeepSeek、Qwen等），通过适配层切换
- **长文本处理**：竞赛论文可能 3000-10000 词，需要分段处理 + 跨段上下文窗口
- **引用校验**：结合 Google Scholar / Semantic Scholar API 做引用真实性校验

### 4.3 关键功能模块（v1.3 新增 Calibration Engine）

| 模块 | 功能 | 优先级 |
|------|------|--------|
| Rubric 模板库 | 预置 20+ 常见竞赛的评分标准模板 | 🔴 高 |
| 版本对比 | 多轮修改的 diff 视图 | 🔴 高 |
| **Calibration Engine** | **批量特征提取 + 差异分析 + 校准建议生成** | **🔴 高（Phase 2）** |
| 导师面板 | 人工导师可覆盖/调整 AI 建议 | 🟡 中 |
| 批量处理 | 一次上传多篇，批量审查 | 🟡 中 |
| 校准日志看板 | 追踪各赛事配置的历史演化和效应量趋势 | 🟡 中 |
| 学习仪表盘 | 追踪学生写作能力的成长曲线 | 🔵 低 |

## 五、风险与边界

| 风险 | 应对策略 |
|------|----------|
| **学生过度依赖 AI** | 系统只做批注不改写；导师最终确认；保留完整的审查日志 |
| **AI 产生幻觉引用** | 所有引用建议必须经过 API 校验；虚假引用标注为 Critical Issue |
| **不同竞赛要求冲突** | 为每个竞赛维护独立 rubric 模板，避免通用化建议 |
| **学生个人 voice 丧失** | Agent 6 专门守护原创性；任何风格建议标注为可选 |
| **小样本校准偏差** | 样本量 < 10 时降低 confidence，标注「仅供参考」；< 5 时不做校准 |
| **评委偏好年度漂移** | 追踪各赛季的 Spearman ρ 趋势；连续两赛季 ρ < 0.6 触发全量重校准 |
| **配置过拟合** | 外部获奖文章作为独立验证集；不做纯粹的数据拟合，保持人工审核环节 |

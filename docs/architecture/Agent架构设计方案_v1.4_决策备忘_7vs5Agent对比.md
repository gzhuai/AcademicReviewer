## 二、量化指标对比

| 维度 | 7 Agent | 5 Agent | 差值 | 判决 |
|------|---------|---------|------|------|
| **每篇 Token 消耗** | ≈ 66K | ≈ 58K | -12% | 差值不到 ¥0.30，**无实际意义** |
| **每篇成本** | ≈ $0.26 (¥1.90) | ≈ $0.22 (¥1.60) | -¥0.30 | 月审 100 篇仅差 ¥30 |
| **端到端延迟** | ≈ 17s | ≈ 13s | -4s | 对异步批处理场景**无感知差异** |
| **LLM 调用次数** | 7 次 | 5 次 | -2 次 | 减少 API 故障面，**有意义** |
| **调度轮次** | 3 轮 | 2 轮 | -1 轮 | 减少一轮串行等待，**有意义** |

> **结论：Token 成本差异可以忽略。7→5 的真正价值不在省钱，在工程层面。**

## 三、审查质量对比（核心战场）

### 3.1 粒度对比

| 审查维度 | 7 Agent 粒度 | 5 Agent 粒度 | 差异 |
|----------|-------------|-------------|------|
| **评分标准解析** | 独立 Agent，专注 | 独立 Agent，专注 | 无 |
| **结构骨架** | 独立 Agent，专注 | 独立 Agent，专注 | 无 |
| **论证深度** | 独立 Agent，专注 | 独立 Agent，专注 | 无 |
| **写作风格** | 独立 Agent | 合并到 A4 | 有讨论空间 |
| **语言润色** | 独立 Agent（唯一可改写） | 合并到 A4 | 有讨论空间 |
| **引用完整性** | 独立 Agent | 合并到 A5 | 有讨论空间 |
| **原创性/AI痕迹** | 独立 Agent | 合并到 A5 | 有讨论空间 |

### 3.2 合并点①深度分析：A4 (Style) + A7 (Language) → Language & Style

**优势**：
- 两个维度在语言表达这条连续谱上自然衔接。语法错误 → 表达不当 → 风格问题，没有硬边界，一个 Agent 能整体把握
- 减少一次全文传输（省 8K token）
- 学生看到的反馈更连贯：不用在"风格问题"和"语法问题"之间来回跳

**劣势/风险**：
- A7 在你的设计中是唯一有「改写权限」的 Agent。合并后需要严格约束：语法修正可改写；风格建议仅批注。这条分界线必须在 prompt 里用铁律写死
- 如果 prompt 没写好，容易出现风格建议侵入式改写（例："建议改为主动语态" → 直接帮你改了整段）
- 学生主权的最重要防线在合并后变薄了

**缓解措施 — A4 改写权限边界（铁律）**：

| 操作类型 | 权限 | 示例 |
|----------|------|------|
| 拼写错误 | ✅ 可改写 | `recieve` → `receive` |
| 语法错误 | ✅ 可改写 | 主谓一致、时态、冠词 |
| 标点修正 | ✅ 可改写 | 逗号、分号、引号规范 |
| 句式选择 | ❌ 仅批注 | "此句可考虑拆分" |
| 语态分布 | ❌ 仅批注 | "建议适当使用主动语态" |
| 用词偏好 | ❌ 仅批注 | "此处可考虑更精确的术语" |
| 段落节奏 | ❌ 仅批注 | "建议增加一个过渡句" |

**输出格式要求**：改写内容用 `[REWRITE]` 标记，批注用 `[SUGGEST]` 标记。

### 3.3 合并点②深度分析：A5 (Citation) + A6 (Originality) → Academic Integrity

**优势**：
- 概念内聚度高：引用缺失 + 抄袭 + AI痕迹 = 学术诚信的不同表现面。现实中三者经常联动：AI 生成引用 → 既是 citation 问题也是 originality 问题；大段未标注引用 → 既是 citation 也是抄袭。一个 Agent 能交叉发现这类复合问题
- 减少一次全文传输（省 8K-10K token）

**劣势/风险**：
- 实现路径不同：引用检查是规则引擎（cite↔ref 配对、格式正则匹配），原创性检查是语义比对（向量相似度、AI痕迹识别）。一个 prompt 管两种逻辑 → prompt 复杂度上升
- 但比合并 A2+A3 的"结构+论证"要安全得多，因为引用和原创的判断方式更互补而非冲突

**缓解措施 — A5 内部两条独立执行路径**：

| 子任务 | 名称 | 驱动方式 | 输出 |
|--------|------|----------|------|
| 5a | Citation Check | 规则引擎 | cite↔ref 配对表、格式错误清单、疑似虚假引用标记 |
| 5b | Originality Check | 语义比对 | 段落相似度、AI痕迹评分、voice 失真警告 |

### 3.4 不会合并的核心三 Agent

无论 7 还是 5，以下三个 Agent 从不合并：

| Agent | 不合并的根本原因 |
|-------|-----------------|
| **A1 Rubric** | 它是配置层，不是审查层。生命周期不同，缓存后一笔复用 |
| **A2 Structure** | "先看骨架、后看血肉"是设计哲学根基。结构检查是二元判断（有/无），论证是连续评估（好/坏），混在一起会互相污染 |
| **A3 Argument** | 它是 L2 赛事定制的核心载体，也是 Calibration Engine 校准的重中之重。独立意味着调校时只改这一个，不影响其他 Agent |

## 四、工程维度对比

| 维度 | 7 Agent | 5 Agent | 分析 |
|------|---------|---------|------|
| **首次开发量** | 高（写 7 套 prompt + 7 个调度逻辑） | 中（写 5 套，其中 2 套更复杂） | 5 Agent 单 prompt 更重，但总量少 |
| **Prompt 单条复杂度** | 低（每个职责单一，prompt 精悍） | 中（A4 和 A5 的 prompt 需要分内部子任务） | 7 版本更容易写出高质量 prompt |
| **调试** | 简单（问题定位精确到单一维度） | 简单到中等（A4/A5 内部需子任务定位） | 微幅差距 |
| **版本迭代** | 灵活（改风格审查不影响语言润色） | 一般（改 A4 影响两个维度，需要回归测试） | 7 版本迭代更安全 |
| **L0/L1/L2 分层** | 清晰 | 基本不受影响（两个合并点都在 L0/L1 层） | — |
| **Calibration Engine 兼容** | 完全兼容 | 完全兼容（校准对象是配置文件不是 Agent） | 无差异 |
| **扩展新赛事类型** | 需要改动 A3 + A4 | 需要改动 A3 + A4 | 无差异 |

## 五、运营维度对比

| 维度 | 7 Agent | 5 Agent | 分析 |
|------|---------|---------|------|
| **反馈报告结构** | 7 维度评分，颗粒度最细 | 5 维度评分，合并后维度略粗 | 7 版本诊断面板更精确 |
| **学生能看出问题归属吗** | 语法和风格分开列 | 合并列在"语言与表达"下，需内部标记来源 | 需确保报告内部区分 |
| **导师覆盖 AI 建议** | 逐 Agent 覆盖，精准 | 合并 Agent 内需子任务级粒度 | 微幅差异 |
| **API 故障容错** | 7 次调用 → 7 个潜在故障点 | 5 次调用 → 降低 28% 故障面 | 5 版本更稳 |

## 六、开发路线反思

v1.2/v1.3 文档推荐的策略是"先开发 7 Agent → Phase 3 合并为 5 Agent"，但这里有一个现实问题：

```
═══════════════════════════════════════════════════════════════
路线 1（文档原推荐）：先开发 7 → 后合并为 5
───────────────────────────────────────────────────────────────
开发 7 Agent：    ████████████████████████████████  (100%)
合并 A4+A7：      ████████                        (+20% 重构)
合并 A5+A6：      ████████                        (+20% 重构)
───────────────────────────────────────────────────────────────
总投入：          140%

路线 2（直接开发 5）：一步到位
───────────────────────────────────────────────────────────────
直接开发 5 Agent： ██████████████████████████████  (90%)
───────────────────────────────────────────────────────────────
总投入：          90%

结论：如果你最终目标是 5 Agent，直接开发 5 比分两步走节省约 36% 总投入。
═══════════════════════════════════════════════════════════════
```

但路线 1 有一个无法替代的价值：在 7 Agent 阶段亲眼看到 A4 和 A7 各自独立的产出，从而用真实数据验证合并是否合理。

## 七、决策矩阵

### 你应该选择 **7 Agent** 如果：

| 条件 | 权重 |
|------|------|
| 你对质量的要求 > 对速度的要求（你是辅导机构，精度是你的护城河） | 最高 |
| 你希望反馈报告给学生的「诊断面板」维度最细（7 个维度 vs 5 个） | 中 |
| 你愿意为更精准的问题定位多写 2 个 Agent 和 1 轮调度 | 中 |
| 你认为「先验证再合并」的路线比「一步到位」更安全 | 参考 |

### 你应该选择 **5 Agent** 如果：

| 条件 | 权重 |
|------|------|
| 你的开发资源有限，需要尽快上线可用版本 | 最高 |
| 你认为语法/风格合在一起不影响反馈质量（实践中确实边界模糊） | 中 |
| 你认为引用/原创合在一起能协同发现复合问题 | 中 |
| 你愿意接受多花一点 prompt 工程复杂度（A4/A5 的内部子任务设计） | 参考 |
| 你需要更少的 API 调用次数来降低模型故障面 | 参考 |

## 八、最终建议

```
╔══════════════════════════════════════════════════════════════╗
║                                                              ║
║   直接开发 5 Agent，但 A4 和 A5 内部保留子任务边界。          ║
║                                                              ║
║   理由：                                                      ║
║   1. Token 和延迟差异可忽略，真正决定效率的是工程复杂度        ║
║   2. 合并的两个点（A4+A7, A5+A6）标记为"✅ 可合并"            ║
║      ——不是勉强合并，它们在概念上是自然内聚的                  ║
║   3. 节省 2 次 API 调用 = 减少 28% 的外部服务故障风险         ║
║   4. 分两步走的总成本是 140%，一步到位是 90%                  ║
║   5. 以后如果发现需要拆分，只需 1 个 Agent 的拆分量，         ║
║      远比合并 2 个 Agent 容易                                 ║
║                                                              ║
║   唯一前提：A4 的 prompt 里必须用铁律划清"改写权限边界"        ║
║                                                              ║
╚══════════════════════════════════════════════════════════════╝
```

**一句话总结**：7 Agent 是「规格最高的方案」，5 Agent 是「性价比最高的方案」。在你的辅导机构场景下——月均 100 篇的量级 + 对反馈精度的要求——5 Agent 在"够用"和"够精"之间取了最优解。省下的 2 个 Agent 不是质量妥协，而是让两个天然内聚的职责合到同一个 context window 里，反而可能产生更好的协同效果。

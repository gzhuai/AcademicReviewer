# 技术方案与执行计划 v1.6

> **关联文档**：Agent 架构 v1.5（5 Agent 规范）、v1.4（决策备忘）、v1.3（双管线+Calibration Engine）

## 一、技术选型

| 层级 | 选型 | 理由 |
|------|------|------|
| **后端框架** | Python 3.12 + FastAPI | AI/LLM 生态最成熟，异步原生支持 |
| **LLM 引擎** | 多模型抽象层，支持：GPT-5.5 / Gemini 3.1 / DeepSeek-v4-pro / GLM-5.1 | 测试对比后择优固定 |
| **LLM 调用库** | 直接 HTTP API（不用 LangChain/LlamaIndex） | 降低依赖复杂度，无维护团队场景更安全 |
| **数据库** | SQLite（通过 SQLAlchemy） | 零维护，单文件，够用到月均 10,000 篇以下 |
| **异步任务** | asyncio + FastAPI BackgroundTasks | 简单够用，避免引入 Celery/Redis 的运维成本 |
| **向量存储** | ChromaDB（嵌入式） | 用于 A5 原创性检查，不需要独立向量数据库服务 |
| **文档解析** | python-docx + PyPDF2 + markdown | 覆盖常见竞赛提交格式 |
| **前端（后期）** | 可选：Gradio（最简）/ Next.js（正式） | Phase 4 再决定 |
| **部署** | 先本地单机 `uvicorn`，Phase 4 上云 | 最小化运维负担 |

## 三、核心模块设计

### 3.1 LLM 多模型抽象层

```python
# app/llm/base.py — 抽象接口
from abc import ABC, abstractmethod

class LLMAdapter(ABC):
    """所有 LLM 适配器的统一接口"""
    
    @abstractmethod
    async def chat(self, system_prompt: str, user_message: str, 
                   temperature: float = 0.3, max_tokens: int = 4096) -> str:
        ...
    
    @abstractmethod
    def model_name(self) -> str:
        ...
```

每个实现类只需封装各自的 HTTP API：

| Adapter | 端点 | 鉴权方式 |
|---------|------|----------|
| `OpenAIAdapter` | `https://api.openai.com/v1/chat/completions` | Bearer Token |
| `GeminiAdapter` | `https://generativelanguage.googleapis.com/v1beta/...` | API Key |
| `DeepSeekAdapter` | `https://api.deepseek.com/v1/chat/completions` | Bearer Token |
| `GLMAdapter` | `https://open.bigmodel.cn/api/paas/v4/chat/completions` | API Key |

**选择逻辑**：通过环境变量 `LLM_PROVIDER` 或单篇审查请求参数动态切换。每个模型一个类，互不耦合。

### 3.2 Agent 基类

```python
# app/agents/base.py
class BaseAgent:
    def __init__(self, llm: LLMAdapter, prompt_template_path: str):
        self.llm = llm
        self.prompt_template = self._load_prompt(prompt_template_path)
    
    async def run(self, **kwargs) -> dict:
        """组装 prompt → 调用 LLM → 返回结构化结果"""
        ...
```

五个 Agent 各自继承 `BaseAgent`，只负责：
1. 加载自己的 prompt 模板
2. 将结构化输入（文稿段落、配置 JSON）拼入 prompt
3. 调用 LLM
4. 解析 LLM 输出为结构化 dict

### 3.3 Orchestrator（调度器）

```python
# app/orchestrator.py
class Orchestrator:
    async def review(self, submission_path: str, competition: str) -> ReviewReport:
        # 1. 解析文稿
        doc = parse_document(submission_path)
        
        # 2. 加载竞赛配置
        comp_type = lookup_competition_type(competition)
        configs = load_configs(comp_type)
        
        # 3. Round 1: 并行执行 A1, A2, A3
        r1_results = await asyncio.gather(
            self.a1.run(rubric_config=configs.rubric),
            self.a2.run(document=doc, schema=configs.structure_schema),
            self.a3.run(document=doc, evidence_config=configs.evidence_patterns),
        )
        
        # 4. Round 2: 并行执行 A4, A5（可接收 R1 的段落标记）
        r2_results = await asyncio.gather(
            self.a4.run(document=doc, style_guide=configs.style_guide, 
                        focus_sections=r1_results.focus_sections),
            self.a5.run(document=doc, citation_rules=configs.citation_rules,
                        submissions_db=self.history_db),
        )
        
        # 5. 聚合生成审查报告
        return aggregate_report(r1_results, r2_results, configs)
```

### 3.6 校准引擎 + 教师经验注入

校准引擎是纯统计管道（零 LLM 调用），通过 Cohen's d 效应量分析获奖作品与失败作品在 16 个文本特征上的差异，自动生成配置文件 diff，供教师审核后生效。

在此基础上，新增强教师经验注入通道——将教师多年辅导经验以文档形式注入校准管道，与统计发现交叉验证。

#### 3.6.1 教师经验文档的三种输入模式

系统自动判定文档格式，采用不同解析策略：

| 模式 | 触发条件 | 解析方式 | 教师门槛 |
|------|----------|----------|----------|
| **Mode 3** 全结构化 | 文档包含 `### [pattern] feature_name` 标记 | 正则精确解析（零成本） | 需学习标记语法 |
| **Mode 2** 半结构化 | 文档包含 `## 获奖经验` / `## 常见问题` 等已知章节标题 | 章节路由 + LLM 辅助拆分条目 | 极低，只需写章节标题 |
| **Mode 1** 纯自由文本 | 以上均不满足 | 全文 LLM 语义解析 | 零门槛 |

#### 3.6.2 Mode 2 支持的章节标题体系

教师只需写以下任一类标题，标题下方自由撰写内容：

| 章节标题系（任一即可） | 映射标注类型 | 用途 |
|---|---|---|
| `## 获奖经验` / `## 获奖规律` / `## 成功案例总结` / `## 高频特征` / `## 晋级规律` / `## 获奖论文的共同特征` | `pattern` | 观察到的获奖模式 |
| `## 常见问题` / `## 常见错误` / `## 易踩的坑` / `## 学生通病` / `## 常见扣分点` / `## 典型问题` | `pitfall` | 常见陷阱与扣分项 |
| `## 高分信号` / `## 晋级信号` / `## 决赛特征` / `## 评审偏好` / `## 加分项` / `## 评委看重什么` | `signal` | 高分信号与评审偏好 |
| `## 教学方法` / `## 辅导策略` / `## 训练方法` / `## 写作建议` / `## 教学心得` / `## 备考策略` | `strategy` | 教学策略与方法建议 |
| `## 评分标准解读` / `## 评分细则` / `## 评审规则理解` | `rubric` | 评分标准解读 |

#### 3.6.3 解析架构

```
教师文档 → detect_mode()
  ├─ Mode 1 (纯自由)     → LLM 全量语义解析 → 结构化 ExpertAnnotation[]
  ├─ Mode 2 (半结构化)   → 章节标题路由 → 按章节拆段 → LLM 辅助提取条目
  └─ Mode 3 (全结构化)   → expert_annotator.py 正则精确解析（现有逻辑，不变）

特征映射阶段（所有模式共用）:
  ExpertAnnotation[] → FEATURE_KEYWORDS 关键词匹配 → mapped_features[]
  （此阶段为确定性本地匹配，不依赖 LLM，确保特征归属准确）
```

#### 3.6.4 LLM 调用策略

- **仅 Mode 1/2 调用 LLM**，Mode 3 不需要
- 一份教师文档只调 **1 次 LLM**（Mode 2 下多章节合并为 1 次调用）
- 输出格式为严格的 JSON schema，由 `chat_json()` 自动解析
- **特征映射环节始终走本地关键词匹配**，避免 LLM 幻觉导致特征归属错误
- 使用项目统一的 `LLMAdapter` 接口，与 Agent 共用同一模型配置

#### 3.6.5 统一入口

```python
# app/calibration/expert_annotator.py  — 新增统一入口
def unify_parse(file_path: str, llm: LLMAdapter | None = None) -> ExpertInsights:
    """自动检测文档格式，选择最优解析策略"""
```

引擎侧 `run_calibration(expert_doc_paths=[...])` 调用此入口，对上游完全透明。

## 四、API 设计（FastAPI）

```
POST /api/v1/review
  - 上传稿件 + 指定竞赛
  - 返回审查报告（JSON）
  - 同步等待（单篇 ≈ 15s）或返回 task_id 异步轮询

GET /api/v1/review/{task_id}
  - 查询异步审查结果

GET /api/v1/competitions
  - 返回支持的竞赛列表

GET /api/v1/reviews/history?competition=xxx&student=xxx
  - 历史审查记录查询

POST /api/v1/config/reload
  - 热加载配置文件（新增/修改竞赛时的运维接口）
```

MVP 阶段只需要 `POST /api/v1/review`（同步版）。

## 七、降低维护成本的设计原则

鉴于无专业维护开发团队，以下原则贯彻整个技术方案：

1. **零外部服务依赖**：SQLite + ChromaDB 都是嵌入式，不需要安装数据库、消息队列、向量数据库等独立服务。整个系统只需要 `pip install` + 一个 `.env` 文件。

2. **纯函数 Agent**：每个 Agent 是无状态的，输入配置 + 文稿 → 输出 dict。不依赖全局状态，方便单独测试和替换。

3. **配置文件即文档**：JSON 配置文件本身是 self-documenting 的，修改竞赛参数不需要改代码。新增竞赛 = 新增一个 JSON 文件 + 注册到 `competition_registry.json`。

4. **最小代码量**：不用 LangChain（太重）、不用 Celery（需要 Redis）、不用 Docker（MVP 不需要）。`run.py` 一行启动。

5. **日志即运维**：所有审查过程写结构化日志，出问题时看日志就能定位是哪个 Agent + 哪段文稿 + 哪个 LLM 调用失败。

## 八、关键风险

| 风险 | 概率 | 缓解 |
|------|------|------|
| GPT-5.5 / Gemini 3.1 模型尚未正式发布或 API 不可用 | 中 | 抽象层设计确保随时切换；先用已确定的模型（DeepSeek-v4-pro / GLM-5.1）起步 |
| A4 改写越界（风格建议变成实际改写） | 高 | Prompt 铁律 + Phase 1 测试中重点验证；出问题只影响 A4 一个 Agent |
| 长文档超 token 限制 | 中 | 分段送入 Agent；学术论文通常 3K-10K 词，在上下文窗口内 |
| 模型 API 不稳定导致审查中断 | 中 | 重试机制（3 次）+ 降级到备选模型 |
| 多模型 API 兼容性不一致 | 低 | 所有 adapter 实现同一接口，差异在 adapter 内部消化 |

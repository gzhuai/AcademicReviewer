# 学术竞赛文稿审查智能体 —— 完整设计方案 v1.2

---

## 一、核心定位与设计原则

### 1.1 核心定位

这个系统的本质是一个 **"AI 学术导师 + 质量门禁"**，而非代写工具。它的核心价值在于：

> 在不替代学生思考的前提下，系统性地发现文稿中的问题，并以**可操作的反馈**指导学生自主改进。

### 1.2 设计原则（五项铁律）

| 原则 | 说明 |
|------|------|
| **学生主权** | AI 只做批注和建议，绝不直接改写正文，确保作品仍为学生原创 |
| **分层审查** | 从宏观（论点结构）到微观（语法拼写），逐层把关，避免挂一漏万 |
| **竞赛对齐** | 每次审查都与具体竞赛的评分标准（rubric）严格对标 |
| **可解释性** | 每个建议必须附带理由和评分维度，让学生"知其所以然" |
| **迭代闭环** | 支持多轮提交→审查→修改→再审，跟踪质量收敛趋势 |

---

## 二、系统架构：Multi-Agent 协作体系

采用 **编排器 + 专家智能体池（Orchestrator + Agent Pool）** 架构：

```
┌──────────────────────────────────────────────────────────┐
│                     Orchestrator                         │
│              (任务调度 / 结果聚合 / 质量报告)               │
└──────┬───────┬───────┬───────┬───────┬───────┬──────────┘
       │       │       │       │       │       │
   ┌───▼──┐┌──▼───┐┌──▼───┐┌──▼───┐┌──▼───┐┌──▼──────┐
   │Rubric││Logic ││Argue ││Style ││Citation││Original│
   │Parser││&Flow ││&Evid ││&Voice││Checker││  Check  │
   └──────┘└──────┘└──────┘└──────┘└───────┘└─────────┘
```

### 2.1 七个专业 Agent 的职责

#### Agent 1: Rubric Parser（评分标准解析器）
- **输入**：竞赛官方 rubric（PDF/图片/文本）
- **输出**：结构化的评分维度树 + 权重 + 关键词
- **能力**：将模糊的评分描述（如 "demonstrates critical thinking"）转化为可检查的具体指标

#### Agent 2: Structure & Logic（结构与逻辑审查）
- 检查论文整体骨架：引言→文献综述→方法→论证→结论的完整性
- 检测段落间的逻辑衔接（是否存在跳跃、重复、缺失环节）
- 评估 thesis statement 的清晰度和全文一致性

#### Agent 3: Argument & Evidence（论点与证据审查）
- 核查每个 claim 是否有充分 evidence 支撑
- 识别逻辑谬误（因果倒置、稻草人、以偏概全等）
- 评估证据来源的权威性和时效性

#### Agent 4: Writing Style & Voice（写作风格与语气）
- 检查写作风格规范性（避免口语化、情绪化表达）
- 术语使用的一致性
- 被动语态/主动语态的合理分布
- 是否符合目标竞赛的预期风格：
  - 学术-技术型（如 ISEF）：干练精确，被动语态可接受
  - 学术-思辨型（如 John Locke）：优雅有力，展示哲学思辨魅力
  - 商务-金融型（如 WGHS）：简洁果断，主动语态优先，数据驱动叙事

#### Agent 5: Citation & Format（引用与格式审查）
- 检查所有引用是否完整（有 cite 必有 reference，反之亦然）
- 验证引用格式（APA / MLA / Chicago 等）的一致性
- 检测可疑的 AI 生成引用（虚假文献检测）

#### Agent 6: Originality Guardian（原创性守护）
- 检测潜在抄袭/过度借鉴段落（与内部历史稿件库比对）
- 识别大段 AI 生成痕迹（语气突变、模板化结构）
- 确保学生个人 voice 得以保留

#### Agent 7: Language Polishing（语言精修）—— 唯一允许改写的 Agent
- 语法、拼写、标点修正
- 中式英语 → 地道表达的建议
- 长难句拆分建议
- **严格以批注模式工作，不直接改动原文**

### 2.2 Agent 配置粒度与复用策略

采用三层分类法管理 Agent 的赛事配置成本：

| 层级 | 含义 | 运营成本 |
|------|------|----------|
| **L0 完全共用** | 所有赛事用同一套，零配置 | 无 |
| **L1 共用引擎 + 赛事配置文件** | 核心逻辑共用，通过一份配置文件切换赛事 | 低（新赛事 ≈ 上传一份文件） |
| **L2 赛事类型专属定制** | 每个竞赛类型需要专门调校 prompt 和验证逻辑 | 高（新类型 ≈ 独立开发子模块） |

**逐个 Agent 判定与原因分析：**

| Agent | 层级 | 原因 |
|-------|------|------|
| **1. Rubric Parser** | **L1** | 审查引擎通用（读取 rubric → 生成维度树），赛事差异全封装在 rubric 文件里。新赛事只需上传一份官方评分标准 PDF。 |
| **2. Structure & Logic** | **L1** | 段落骨架检查逻辑通用，差异在于结构模板。学术论文用 `Abstract→Introduction→...→Conclusion`，金融报告用 `Executive Summary→Investment Thesis→...→Risk Analysis`。通过维护一份结构 schema 文件切换。 |
| **3. Argument & Evidence** | **L2** | 不同赛事类型对"好论证"的定义截然不同。科研型检查实验数据和统计显著性；思辨型检查逻辑链和哲学引用；金融型检查估值模型合理性和财务数据溯源。同一套审查逻辑无法跨类型复用。**同一赛事类型内的不同竞赛可共享（如 ISEF 和 STS 共用"科研型"）。** |
| **4. Writing Style & Voice** | **L2** | 风格期望差异巨大。学术技术型（ISEF）要求干练精确；学术思辨型（John Locke）要求优雅有力；商务金融型（WGHS）要求简洁果断、数字驱动。需要针对性 prompt engineering 和范文调校。 |
| **5. Citation & Format** | **L1** | 引用校验逻辑通用（cite-reference 配对、格式一致性），差异仅在于格式规则（APA / MLA / Chicago / 金融数据脚注式）。通过格式规则文件驱动。 |
| **6. Originality Guardian** | **L0** | 学术诚信检查不区分赛事——抄袭检测、AI 痕迹识别、大段重复判断逻辑 100% 通用。唯一赛事相关的是"内部历史稿件比对库"，但那只是数据源，引擎本身不变。 |
| **7. Language Polishing** | **L0+L1** | 语法/拼写/标点检查完全通用（L0）。风格微调偏好可配置（L1）：学术模式允许被动语态、长句容忍度高；商务模式建议主动语态、标记过长句子。 |

**关键结论：**

- 7 个 Agent 中，**6 个高度可复用**（L0 + L1），共用底层引擎，新赛事只需配置
- 真正需要赛事类型专属投入的只有 **Agent 3 和 Agent 4**
- Agent 3+4 的定制粒度是**赛事类型**而非单个赛事，同一类型内赛事间可复用
- 若一开始聚焦 1-2 个赛事类型（如主打科研类），Agent 3+4 只需一次投入

### 2.3 竞赛类型五分类

系统将常见国际竞赛归为五大类型，每种类型驱动不同的 L2 审查逻辑：

| 类型 | Agent 3 审查重点 | Agent 4 风格标准 | 典型竞赛 |
|------|-----------------|-----------------|----------|
| **科研型** | 实验设计、数据显著性、可复现性、变量控制 | 学术-技术：干练精确、被动语态可接受 | ISEF, Regeneron STS, ISSDC |
| **思辨型** | 逻辑链完整性、哲学引用深度、反驳力度 | 学术-思辨：优雅有力、展示个人 voice | John Locke, Marshall Society |
| **社科型** | 一手调研方法论、案例分析深度、跨文化视角 | 学术-社科：调研叙事、平衡客观与洞察 | CTB, YOC, YIC |
| **历史型** | 一手史料引用、史学视角、contextualization | 学术-史学：史实精准、时代语境还原 | NHD, The Concord Review |
| **金融型** | 估值模型合理性、财务数据溯源、风险分析框架、图表质量 | 商务-金融：简洁果断、数据驱动、主动语态 | SIC, WGHS, YIS |

**金融型竞赛的特殊性：**

金融投资报告（如 WGHS、SIC、YIS）与学术论文存在本质差异，是五大类型中复用度最低的一类：

| 维度 | 传统学术论文 | 金融投资报告 |
|------|-------------|-------------|
| **写作目的** | 论证学术观点 | 说服投资决策 |
| **核心骨架** | Intro → Lit Review → Method → Results → Discussion | Executive Summary → Investment Thesis → Industry Analysis → Valuation → Portfolio → Risk Analysis |
| **证据类型** | 学术引用、实验数据、一手史料 | 财务数据、估值模型（DCF/Comparables）、SEC 文件、Bloomberg 数据 |
| **写作风格** | 学术英语，被动语态可接受，hedging language | 商务英语，短句优先（15-20词），主动语态，明确判断 + 证据支撑 |
| **图表角色** | 辅助说明 | **核心叙事载体**（K线图、财务模型、敏感性分析表） |
| **引用体系** | APA/MLA/Chicago 正式引用 | 脚注式数据来源标注（如 SEC 10-K FY2024, Bloomberg Terminal），精确但不拘泥于格式 |
| **关键风险** | 逻辑谬误、证据不足 | 估值假设不合理、数据来源缺失、风险对冲策略缺失 |

**新赛事接入工作量分布（以已完成第一个类型为基础）：**

```
同一赛事类型内接入第二个竞赛：
████████░░░░  15%  上传 Rubric + 微调配色 → 约 1-2 天

新接入一个赛事类型（如已有科研型，新增金融型）：
████████████████████████░░  70%
  ├── Agent 3 论证逻辑开发（估值检查、风险框架、数据溯源）   40%
  ├── Agent 4 风格模板开发（商务写作风格指南）            20%
  ├── Agent 2 结构 schema（金融报告骨架定义）             5%
  └── Agent 1+5 Rubric + 引用规则配置                     5%

之后同类型的第二个金融竞赛：
████░░░░░░░░  10%  纯配置工作 → 约半天
```

### 2.4 精简方案与取舍分析

本章探讨 7 Agent 架构是否可精简，以及精简后的成本、性能和质量权衡。

#### 2.4.1 Token 成本与延迟测算

以一篇 **5000 词**的典型竞赛论文为基准，对 7 Agent 架构做定量评估：

| Agent | 输入 Token | 输出 Token | 输入内容 |
|-------|-----------|-----------|----------|
| 1. Rubric Parser | 3K | 1K | 仅 rubric 文本（不传全文） |
| 2. Structure & Logic | 8K | 2K | 全文 + 结构 schema |
| 3. Argument & Evidence | 8K | 3K | 全文 + rubric 维度 |
| 4. Writing Style & Voice | 8K | 1.5K | 全文 |
| 5. Citation & Format | 8K | 1.5K | 全文 + ref 列表 |
| 6. Originality Guardian | 10K | 1K | 全文 + 比对片段 |
| 7. Language Polishing | 8K | 3K | 全文 |

| 汇总 | 数值 |
|------|------|
| 总输入 Token | ≈ 53K |
| 总输出 Token | ≈ 13K |
| **单次审查总消耗** | **≈ 66K tokens** |
| 按 GPT-4o 价格估算 | **≈ $0.26/篇（约 ¥1.90）** |
| 月均审查 100 篇 | ≈ ¥190 |
| 年化成本 | ≈ ¥2,280 |

**结论：Token 成本在日常运营量级下几乎可忽略。真正的优化价值不在于省钱，而在于：① 减少端到端延迟；② 降低工程复杂度；③ 提高单次响应的可靠性和可调试性。**

**延迟对比：**

```
7 Agent（当前设计，分轮并行）:
Round 1: A1 ████░░ + A2 ██████░░ + A3 ████████░░ → 最长 6s
Round 2: A4 ██████░░ + A5 █████░░ + A7 ████████░░ → 最长 5s
Round 3: A6 ████████░░ → 6s
────────────────────────────────────────────────
总计 ≈ 17s

串行模式（如果不并行）:
A1→A2→A3→A4→A5→A6→A7 → ≈ 35s ⚠️
```

#### 2.4.2 Agent 合并可行性矩阵

逐个分析各 Agent 对之间的合并可行性：

| Agent X | 与 Agent Y 合并 | 判定 | 理由 |
|---------|----------------|------|------|
| A1 Rubric | A2/A3/A4/A5/A6/A7 | ❌ 不可 | Rubric 解析是前置任务，输出供所有下游消费，本质是"配置层"而非"审查层" |
| A2 Structure | A3 Argument | ⚠️ 可但存疑 | 结构看"骨架有没有"（完整性），论证看"肉好不好"（深度）。合并后 prompt 臃肿，容易出现"重视结构忽略论证"的审查偏差 |
| A2 Structure | A4 Style | ❌ 不可 | 粒度分裂——你设计里最重要的约束就是宏微观分离，合并会破坏这一原则 |
| A3 Argument | A4 Style | ❌ 不可 | 一个审逻辑，一个审表达，混在一起会互相污染 |
| **A4 Style** | **A7 Language** | **✅ 可合并** | 风格和语言是连续谱——语法错误→表达不当→风格问题没有硬边界。代价：A7 是唯一有"改写权限"的 Agent，合并后需严格约束"改写建议仅限于语法层面" |
| **A5 Citation** | **A6 Originality** | **✅ 可合并** | 两者同属"学术诚信"范畴——引用缺失和抄袭本质上是不诚实的不同表现。代价：实现方式不同（规则匹配 vs 语义比对），合并后 prompt 复杂度上升但概念内聚 |
| A5 Citation | A1 Rubric | ❌ 不可 | 引用检查需要全文，与 rubric 解析的生命周期不同 |
| A6 Originality | 其他 | ⚠️ 可独立 | 原创性检查涉及外部比对（历史稿件库），依赖前两轮结果，建议保留独立性 |

**可行合并点只有两个**：
- **合并点①**：Agent 4 (Writing Style) + Agent 7 (Language Polishing) → 统管"语言表达层"
- **合并点②**：Agent 5 (Citation & Format) + Agent 6 (Originality Guardian) → 统管"学术诚信层"

#### 2.4.3 方案 A：5 Agent（推荐）

```
┌──────────────────────────────────────────────┐
│               Orchestrator                   │
└──┬──────┬──────┬──────┬──────┬──────────────┘
   │      │      │      │      │
   ▼      ▼      ▼      ▼      ▼
┌────┐ ┌────┐ ┌────┐ ┌────┐ ┌─────────┐
│ A1 │ │ A2 │ │ A3 │ │ A4 │ │   A5    │
│Rub │ │Str │ │Arg │ │Lang│ │Academic │
│Par │ │&Log│ │&Ev │ │&Sty│ │Integrity│
└────┘ └────┘ └────┘ └────┘ └─────────┘
                         ↑         ↑
                    合并 4+7    合并 5+6
```

| Agent | 名称 | 合并来源 | 职责 |
|-------|------|----------|------|
| A1 | Rubric Parser | 原 A1 | 不变 |
| A2 | Structure & Logic | 原 A2 | 不变 |
| A3 | Argument & Evidence | 原 A3 | 不变 |
| A4 | Language & Style | 原 A4 + A7 | 语言修正（有改写权限，但限语法层面）+ 风格审查（仅批注模式） |
| A5 | Academic Integrity | 原 A5 + A6 | 引用完整性检查 + 原创性/抄袭检测 |

**调度 Round 结构（从 3 轮压缩至 2 轮）：**

```
Round 1（宏观 + 语言）- 并行:
  ├── Agent 1: Rubric Parser → 生成评分检查清单
  ├── Agent 2: Structure & Logic → 结构完整性报告
  ├── Agent 3: Argument & Evidence → 论证力评估
  └── Agent 4: Language & Style → 语言与风格审查

Round 2（诚信）- 基于 R1:
  └── Agent 5: Academic Integrity → 引用 + 原创性
```

| 指标 | 7 Agent | 5 Agent | 变化 |
|------|---------|---------|------|
| Agent 数量 | 7 | 5 | -28% |
| 调用次数 | 7 次 | 5 次 | -28% |
| 串行延迟（并行后） | ≈ 17s | ≈ 13s | -24% |
| 单次 Token | ≈ 66K | ≈ 58K | -12% |
| 审查质量 | 基准 | **基本无损失** | — |
| 调试复杂度 | 简单 | 简单 | — |
| 扩展灵活性 | 灵活 | 灵活 | — |

**注意事项：**
- A4 合并后必须强制约束：语法修正可改写，风格建议仅批注——这是防止"AI 代写"的最后防线
- A5 需要两条独立的内部执行路径：引用检查走规则引擎（cite↔ref 配对），原创性检查走语义比对

**风险点：**
- A2 (Content Review) 合并了结构和论证，prompt 复杂度显著上升
- 可能存在"重视结构完整性、忽略论证深度"的偏差，因为结构检查（有/无）比论证检查（好/坏）更容易执行
- Prompt 长度增加可能降低 LLM 对细节的关注度（lost-in-the-middle 效应）

#### 2.4.5 方案 C：3 Agent（不推荐）

```
A1: Rubric Parser
A2: Full Content Review（结构 + 论证 + 风格 + 语言）
A3: Academic Integrity（引用 + 原创性）
```

**不推荐理由：**
- A2 的 prompt 将是一个"巨兽"，涵盖四个维度的审查逻辑，难以调试、优化和版本管理
- 与你"分层审查、逐层反馈"的设计初衷背道而驰——你会给学生一个混乱的综合报告，而非结构化的分层建议
- "先别急着改语法，先看论点站不站得住"——这个核心设计原则会被破坏
- Lost-in-the-middle 效应严重：prompt 太长，LLM 倾向于只关注开头和结尾

#### 2.4.6 方案对比总览

```
                       保守 ←────────────────→ 激进

 7 Agent ────── 5 Agent ⭐ ────── 4 Agent ────── 3 Agent
 (v1.1)        (推荐)            (可选)         (不推荐)

 代价/篇：    $0.26          $0.22          $0.20          $0.19
 延迟：       17s            13s            10s            8s
 质量：       基准           基本无损失      有轻微损失      显著损失
 调试：       简单           简单           中等           困难
 扩展：       灵活           灵活           一般           困难
 Round数：    3轮            2轮            2轮            2轮
```

**结论：5 Agent 为最佳平衡点。保留清晰的职责边界，减少 28% 调用次数，几乎不损失审查质量，从 3 轮调度压缩至 2 轮。**

#### 2.4.7 为什么保留不合并的核心 Agent

即便在精简方案中，以下三个 Agent 始终坚持不合并：

| Agent | 不合并的核心原因 |
|-------|-----------------|
| **A1 Rubric Parser** | 一次性前置任务，输出缓存后全局复用。它不是"审查层"而是"配置层"，与审查 Agent 生命周期不同。合并会迫使每次审查都重新解析 rubric，浪费 token |
| **A2 Structure & Logic** | "先看骨架、后看血肉"是整个设计的哲学根基。结构是二元的（有/无框架），论证是连续的（好/坏深度），审查逻辑本质上不同 |
| **A3 Argument & Evidence** | 这是 L2 赛事定制的核心载体。将其独立，意味着为不同赛事类型调校时只改这一个 Agent，不影响结构和语言审查。合并会打乱配置粒度的 L0/L1/L2 分层 |

#### 2.4.8 v1.2 推荐策略

- **开发阶段（Phase 1-2）**：先按 7 Agent 构建，充分验证各个维度的审查质量
- **优化阶段（Phase 3 之前）**：按方案 A 合并为 5 Agent，重点是 A4 (Lang+Style) 和 A5 (Citation+Originality) 的合并
- **保留选项**：4 Agent 方案作为降级策略，在需要严格控延迟的场景下可切换

## 三、工作流设计：五阶段管道

### 阶段流程图

```
输入阶段          审查阶段                        反馈阶段        迭代阶段
┌──────┐    ┌─────────────────────────┐    ┌──────────┐    ┌──────────┐
│ Stage1│───▶│ Stage2: Multi-Agent Review│───▶│ Stage3:  │───▶│ Stage4:  │
│ Intake│    │  Rubric→Structure→Logic    │    │ Feedback │    │ Revision │
│       │    │  →Evidence→Style→Citation  │    │ Report   │    │ Support  │
└──────┘    └─────────────────────────┘    └──────────┘    └──────────┘
                                                               │
                                                          ┌────▼─────┐
                                                          │ Stage5:  │
                                                          │ Final QC │
                                                          │ & Export │
                                                          └──────────┘
```

### Stage 1: 材料摄入与预处理（Intake）

**输入材料清单：**

| 类别 | 材料 | 必须/可选 | 说明 |
|------|------|-----------|------|
| **学生文稿** | 完整英文稿件（.docx / .tex / .md） | ✅ 必须 | 支持带格式文档 |
| **竞赛信息** | 竞赛名称、类别、提交截止日期 | ✅ 必须 | 用于加载对应 rubric 模板 |
| **评分标准** | 官方 rubric / judging criteria | ✅ 必须 | 最好是 PDF 原文，文本也可 |
| **字数要求** | 最低/最高字数限制 | 🟡 建议 | 超字数或字数不足的检测 |
| **格式要求** | 字体、行距、页边距等 | 🟡 建议 | 格式合规自动检查 |
| **往年范文** | 该竞赛往届获奖作品 | 🔵 可选 | 用于风格对标和质量基准 |
| **学生背景** | 年级、英语水平、过往成绩 | 🔵 可选 | 个性化反馈深度 |
| **历史版本** | 该学生此文稿的前序版本 | 🔵 可选 | 用于追踪改进轨迹 |
| **导师备注** | 辅导老师对该文稿的初步判断 | 🔵 可选 | 辅助 AI 聚焦关键问题 |

**预处理操作：**
- 将 .docx/.pdf 转换为结构化 Markdown（保留标题层级）
- 自动分段、分句、分段落类型标注（intro/body/conclusion）
- 统计字数、段落数、引用数等基础指标

### Stage 2: 多 Agent 并行审查

**核心思路：先粗后细，先整体后局部。**

```
Orchestrator 调度顺序:

Round 1（宏观）- 并行执行:
  ├── Agent 1: Rubric Parser → 生成评分检查清单
  ├── Agent 2: Structure & Logic → 结构完整性报告
  └── Agent 3: Argument & Evidence → 论证力评估

Round 2（微观）- 基于 Round 1 结果执行:
  ├── Agent 4: Writing Style → 风格一致性检查
  ├── Agent 5: Citation Check → 引用完整性检查
  └── Agent 7: Language Polish → 语言问题标注

Round 3（综合）:
  └── Agent 6: Originality Guardian → 原创性审查（需要前两轮结果）
```

**关键设计决策：分离审查粒度**

- **宏观看结构**：就算语法完美，论点站不住脚 = 不及格
- **微观查细节**：结构再好，引文格式全错 = 扣分
- **先别急着改语法**，很多学生会因为改语法而产生"已经改好了"的错觉，忽略了更大的结构性问题

### Stage 3: 反馈报告生成

反馈报告不是简单的错误列表，而是分层级的：

```markdown
# 学术竞赛文稿审查报告

## 📊 总体评分预估
| 维度 | 评分 | 权重 | 加权 |
|------|------|------|------|
| 论点与论证 | 6.5/10 | 30% | 1.95 |
| 结构与逻辑 | 7.0/10 | 25% | 1.75 |
| 学术规范 | 8.0/10 | 20% | 1.60 |
| 语言表达 | 7.5/10 | 15% | 1.13 |
| 原创性   | 9.0/10 | 10% | 0.90 |
| **加权总分** | | | **7.33/10** |

## 🔴 Critical Issues（必须修改）
- [ ] Thesis statement 在第2段之后出现偏离...
- [ ] 第4段的 claim 缺少 citation 支撑...

## 🟡 Major Suggestions（强烈建议）
- [ ] 方法论部分缺少对样本量的说明...
- [ ] 讨论部分未回应引言中提出的核心问题...

## 🔵 Minor Improvements（锦上添花）
- [ ] 第3段可考虑增加一个过渡句...
- [ ] 关键词 "sustainability" 出现了12次，建议用同义词替换部分...

## 💡 Competition-Specific Tips
- ISEF 评委通常对实验可复现性特别关注，建议补充...
```

### Stage 4: 学生修改辅助

这个阶段提供的是**不代写的辅助工具**：
- 针对每条 Critical Issue 提供引导性提问（例如："你的实验中，控制组和实验组的关键区别是什么？请尝试在方法部分补充"）
- 提供同义表达库（不直接改句子，而是给 3-5 个可选的表达方向）
- 修改后自动触发增量审查（只审查修改过的段落 + 受影响的上下游段落）

### Stage 5: 终审与导出

- 最终合规检查（字数、引用格式、文件命名等硬性要求）
- 自动生成提交清单（Checklist）
- 导出带批注的 .docx 或 PDF（批注为评论模式，便于人工导师最终把关）

## 四、技术实现建议

### 4.1 推荐的 Prompt 工程策略

每个 Agent 使用**结构化 Prompt 模板**，至少包含：

```
[Role] 你是XXX领域的竞赛评委，拥有15年评审经验
[Context] 这是参加XXX竞赛的高中生论文，竞赛rubric如下...
[Task] 请逐段审查本文的结构逻辑，重点关注...
[Output Format] 请严格按以下JSON格式输出...
[Constraints] 不要重写学生的句子，只指出问题和改进方向
[Examples] 好的反馈 vs 差的反馈示例...
```

### 4.2 与现有 LLM 的适配

鉴于在中国大陆运营，建议考虑：
- **模型层抽象**：同时支持 OpenAI / Claude / 国产模型（DeepSeek、Qwen等），通过适配层切换
- **长文本处理**：竞赛论文可能 3000-10000 词，需要分段处理 + 跨段上下文窗口
- **引用校验**：结合 Google Scholar / Semantic Scholar API 做引用真实性校验

### 4.3 关键功能模块

| 模块 | 功能 | 优先级 |
|------|------|--------|
| Rubric 模板库 | 预置 20+ 常见竞赛的评分标准模板 | 🔴 高 |
| 版本对比 | 多轮修改的 diff 视图 | 🔴 高 |
| 导师面板 | 人工导师可覆盖/调整 AI 建议 | 🟡 中 |
| 批量处理 | 一次上传多篇，批量审查 | 🟡 中 |
| 学习仪表盘 | 追踪学生写作能力的成长曲线 | 🔵 低 |

## 五、风险与边界

| 风险 | 应对策略 |
|------|----------|
| **学生过度依赖 AI** | 系统只做批注不改写；导师最终确认；保留完整的审查日志 |
| **AI 产生幻觉引用** | 所有引用建议必须经过 API 校验；虚假引用标注为 Critical Issue |
| **不同竞赛要求冲突** | 为每个竞赛维护独立 rubric 模板，避免通用化建议 |
| **学生个人 voice 丧失** | Agent 6 专门守护原创性；任何风格建议标注为可选 |

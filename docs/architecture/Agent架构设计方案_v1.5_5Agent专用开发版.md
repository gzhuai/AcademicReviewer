**对 7 Agent 的合并**：
- A4 = 原 A4 (Writing Style) + 原 A7 (Language Polishing)
- A5 = 原 A5 (Citation) + 原 A6 (Originality Guardian)

**Calibration Engine** 位于架构底层，通过分析获奖/失败文章的统计差异（Cohen's d），自动生成配置文件更新建议。教师可将多年辅导经验以文档形式注入校准管道，与统计发现交叉验证，产出更精准的配置优化方案。详见[第八章](#八校准引擎与教师经验注入)。

## 二、五个 Agent 详细规格

### Agent 1: Rubric Parser（评分标准解析器）

| 属性 | 值 |
|------|-----|
| **配置层级** | L1 |
| **输入** | 竞赛官方 rubric（PDF/图片/文本） |
| **输出** | 结构化评分维度树 + 权重 + 关键词 |
| **配置文件** | `rubric_calibrated.json` |
| **校准需求** | ✅ 深 — 维度权重反推、子维度发现 |
| **调度时读取全文？** | ❌ 不读。仅解析 rubric，结果缓存供下游共用 |
| **调度轮次** | Round 1（与 A2/A3 并行） |

核心能力：将模糊评分描述（如 "demonstrates critical thinking"）转化为可检查的具体指标。

---

### Agent 2: Structure & Logic（结构与逻辑审查）

| 属性 | 值 |
|------|-----|
| **配置层级** | L1 |
| **输入** | 全文 + 结构 schema |
| **输出** | 结构完整性报告、段落衔接评估、thesis 一致性评分 |
| **配置文件** | `structure_schema_calibrated.json` |
| **校准需求** | ✅ 中浅 — 二元检查项升级为「存在+质量预期」 |
| **调度轮次** | Round 1（与 A1/A3 并行） |

检查项：
- 引言→文献综述→方法→论证→结论的完整性（学术论文骨架）
- Executive Summary→Investment Thesis→Valuation→Risk Analysis（金融报告骨架）
- 段落间逻辑衔接：是否存在跳跃、重复、缺失环节
- Thesis statement 清晰度与全文一致性

---

### Agent 3: Argument & Evidence（论点与证据审查）

| 属性 | 值 |
|------|-----|
| **配置层级** | L2 — 赛事类型专属 |
| **输入** | 全文 + rubric 维度 |
| **输出** | 论证力评估、逻辑谬误清单、fatal_defects 标记 |
| **配置文件** | `evidence_patterns.json`（按赛事类型） |
| **校准需求** | ✅✅ 最深 — 论证模式反推、fatal_defects 挖掘、每次有新样本即触发 |
| **调度轮次** | Round 1（与 A1/A2 并行） |

检查项：
- 每个 claim 是否有充分 evidence 支撑
- 逻辑谬误识别（因果倒置、稻草人、以偏概全等）
- 证据来源的权威性和时效性
- 依据 `evidence_patterns.json` 中校准后的 fatal_defects 标记「有则必死」硬伤

---

### Agent 4: Language & Style（语言与风格审查）—— 合并 Agent

| 属性 | 值 |
|------|-----|
| **合并来源** | 原 A4 (Writing Style & Voice) + 原 A7 (Language Polishing) |
| **配置层级** | L0（语法/拼写） + L1（风格开关：学术/商务） |
| **输入** | 全文 + style_guide.json + Round 1 中各 Agent 的关注段落 |
| **输出** | 语言修正建议 + 风格批注 + 综合语言诊断报告 |
| **配置文件** | `style_guide.json`（按赛事类型） |
| **校准需求** | ✅ 中 — 风格特征区间统计 |
| **调度轮次** | Round 2（与 A5 并行） |

#### 内部子任务划分：

```
┌─────────────────────────────────────┐
│            Agent 4                  │
│        Language & Style             │
├─────────────────────────────────────┤
│                                     │
│  ┌─────────────────────────────┐   │
│  │  子任务 4a：Language Fix     │   │
│  │  权限：✅ 可改写（有且仅有）    │   │
│  │  范围：拼写、语法、标点        │   │
│  │  输出标记：[REWRITE]          │   │
│  └─────────────────────────────┘   │
│                                     │
│  ┌─────────────────────────────┐   │
│  │  子任务 4b：Style Review     │   │
│  │  权限：❌ 仅批注，绝不改写     │   │
│  │  范围：句式、语态、用词、节奏  │   │
│  │  输出标记：[SUGGEST]          │   │
│  └─────────────────────────────┘   │
│                                     │
└─────────────────────────────────────┘
```

#### 铁律 — 改写权限边界（必须在 prompt 中显式声明）：

| 操作类型 | 权限 | 示例 |
|----------|------|------|
| 拼写错误 | ✅ 可改写 | `recieve` → `receive` |
| 语法错误 | ✅ 可改写 | 主谓一致、时态、冠词 |
| 标点修正 | ✅ 可改写 | 逗号、分号、引号规范 |
| 中式英语修正 | ✅ 可改写 | 仅限语法层面，不改变作者原意 |
| 句式选择 | ❌ 仅批注 | "此句可考虑拆分" |
| 语态分布 | ❌ 仅批注 | "被动语态占比 65%，学术论文建议 40-50%" |
| 用词偏好 | ❌ 仅批注 | "此处可考虑更精确的术语" |
| 段落节奏 | ❌ 仅批注 | "建议在第3段和第4段之间增加一个过渡句" |
| 风格判断 | ❌ 仅批注 | "当前风格偏口语化，学术文体建议更正式的表达" |

#### A4 Prompt 铁律片段（中文版，供开发参考）：

```
你是学术竞赛文稿的语言与风格审查专家。

【改写权限】—— 这是你最重要的约束：
- 你只允许改写以下内容：拼写错误、语法错误、标点错误、中式英语语法修正
- 以下内容你只允许批注、不允许改写：句式选择、语态分布、用词偏好、段落节奏、风格判断
- 违例示例：把"建议用主动语态"直接改写为主动语态句子 → 这是严禁的

【输出格式】：
- 每个改写的句子用 [REWRITE] 标记，包含原文和改写后文本
- 每个风格建议用 [SUGGEST] 标记，仅描述问题和改进方向，不提供改写文本
- 严格分开输出 REWRITE 区和 SUGGEST 区
```

### Agent 5: Academic Integrity（学术诚信审查）—— 合并 Agent

| 属性 | 值 |
|------|-----|
| **合并来源** | 原 A5 (Citation & Format) + 原 A6 (Originality Guardian) |
| **配置层级** | L0 + L1（引用规则文件） |
| **输入** | 全文 + 参考文献列表 + 内部历史稿件库（向量比对） |
| **输出** | 引用完整性报告 + 原创性评分 + 疑似问题标记 |
| **配置文件** | `citation_rules.json` |
| **校准需求** | ❌ 无需校准（规则驱动） |
| **调度轮次** | Round 2（与 A4 并行） |

#### 内部子任务 — 两条独立执行路径：

```
┌───────────────────────────────────────────┐
│              Agent 5                      │
│         Academic Integrity                │
├───────────────────────────────────────────┤
│                                           │
│  ┌───────────────────────────────────┐   │
│  │  子任务 5a：Citation Check         │   │
│  │  驱动：规则引擎                     │   │
│  │  ├── cite ↔ ref 配对检查            │   │
│  │  ├── 引用格式一致性（APA/MLA等）     │   │
│  │  ├── 疑似 AI 幻觉引用检测            │   │
│  │  └── 输出：结构化配对表 + 错误清单   │   │
│  └───────────────────────────────────┘   │
│                                           │
│  ┌───────────────────────────────────┐   │
│  │  子任务 5b：Originality Check      │   │
│  │  驱动：语义比对                     │   │
│  │  ├── 与内部历史稿件库相似度比对      │   │
│  │  ├── AI 生成痕迹检测               │   │
│  │  ├── 大段重复/模板化结构检测        │   │
│  │  ├── 学生个人 voice 保留度评估      │   │
│  │  └── 输出：相似度评分 + 可疑段落标记 │   │
│  └───────────────────────────────────┘   │
│                                           │
└───────────────────────────────────────────┘
```

**实施建议**：子任务 5a 尽量不依赖 LLM，走正则 + 规则引擎；子任务 5b 用嵌入向量做语义相似度，LLM 仅做最终综合判断和报告生成。

## 三、Orchestrator 调度（2 轮制）

```
═══════════════════════════════════════════════════════════
Round 1（宏观层）- 4 个 Agent 并行:
───────────────────────────────────────────────────────────
  A1 Rubric Parser      ← rubric_calibrated.json
      输出：评分维度树 + 检查清单（缓存供下游共用）
  A2 Structure & Logic  ← structure_schema_calibrated.json
      输出：结构完整性报告
  A3 Argument & Evid    ← evidence_patterns.json
      输出：论证力评估 + fatal_defects 标记

  ⚡ 全部并行，最长耗时 ≈ 6s
───────────────────────────────────────────────────────────
Round 2（微观+诚信）- 2 个 Agent 并行（依赖 R1 结果）:
  A4 Language & Style   ← style_guide.json + R1 关注段落
      输出：[REWRITE] 区 + [SUGGEST] 区
  A5 Academic Integrity ← citation_rules.json
      输出：引用报告 + 原创性报告

  ⚡ 全部并行，最长耗时 ≈ 7s
═══════════════════════════════════════════════════════════
总延迟 ≈ 13s（R1 + R2 串行）
```

## 六、竞赛类型七分类支持

系统根据竞赛类型加载不同配置组合。5 Agent 的 L0/L1/L2 分层设计确保了：**新增竞赛类型只影响 A3 和 A4 两个 Agent，而且主要影响的是它们的配置文件**。A1、A2（结构 schema 除外）、A5 完全不受影响。

### 6.1 七分类总览

| 类型 | A3 论证标准 | A4 风格模板 | A2 结构模板 | 典型竞赛 |
|------|------------|------------|------------|----------|
| **科研型** | 实验设计、显著性、可复现性 | 学术-技术：干练精确 | 学术论文标准骨架 | ISEF, STS |
| **科研-进阶型** | 理论贡献度、文献 gap 深度、方法原创性、上限比科研型高一档 | 学术-技术：干练精确 | 学术论文标准骨架 | 丘成桐 |
| **数模型** | 模型假设合理性、求解路径可追溯、灵敏度分析、公式正确性 | 学术-技术：公式友好、图表精确 | 数学建模骨架（问题→假设→符号→模型→求解→检验） | HiMCM, IMMC, MCM |
| **思辨型** | 逻辑链、哲学引用、反驳力 | 学术-思辨：优雅有力 | 学术论文标准骨架 | John Locke, Marshall |
| **时政/经济写作** | 事实引用+多元视角（时政）、理论工具+数据支持（经济） | 评论风：简洁有力有立场 / 学术-经济风 | 思辨论文骨架 | Concord Review, 时政/经济 essay |
| **社科型** | 调研方法、案例深度 | 学术-社科：调研叙事 | 社科报告骨架 | CTB, YOC |
| **历史型** | 一手史料、史学视角 | 学术-史学：史实精准 | 史学论文骨架 | NHD |
| **金融型** | 估值模型、数据溯源、图表 | 商务-金融：简洁果断 | 金融报告骨架 | SIC, WGHS, YIS |
| **商科型** | SWOT/PEST/五力框架应用、方案可行性、ROI 计算 | 商务-提案：说服力优先、clear recommendation | 案例分析骨架（诊断→框架→方案→计划→风险） | FBLA, DECA, BPA |

### 6.2 各类型对 Agent 的冲击面

| 类型 | 受影响 Agent | 改动内容 | 工作量 | 是否需新 L2 |
|------|:---:|------|--------|:---:|
| 科研型 | — | 基线，无额外改动 | 基线 | — |
| 科研-进阶型 | A3 | 科研型 A3 论证标准的阈值提升版 | 低 | ❌ 科研型子类 |
| 数模型 | A3 | 新增公式可追溯性检查、模型假设合理性验证 | 中 | ✅ 新 L2 配置 |
| 思辨型 | — | 基线 | 基线 | — |
| 时政/经济写作 | A3 + A4 | 思辨型 A3 的视角检查扩展 + 经济理论引用验证 | 低 | ❌ 思辨型子类 |
| 社科型 | — | 基线 | 基线 | — |
| 历史型 | — | 基线 | 基线 | — |
| 金融型 | — | 基线 | 基线 | — |
| 商科型 | A3 + A4 | 框架应用检查 + 商业提案风格模板 | 中高 | ✅ 新 L2 配置 |

### 6.3 类别判定规则

当用户提交审查请求时，Orchestrator 根据竞赛名称查表判定类型，加载对应配置：

```
Orchestrator 类型判定逻辑:
  输入: competition_name
  查表: competition_registry.json
  输出: { type, subtype, structure_schema, evidence_config, style_template }
  
示例:
  "ISEF"     → type: "research",        subtype: null
  "丘成桐"    → type: "research",        subtype: "advanced"
  "HiMCM"    → type: "math_modeling",   subtype: null
  "John Locke" → type: "discursive",    subtype: null
  "Marshall"  → type: "discursive",     subtype: "economics"
  "FBLA"     → type: "business_case",   subtype: null
```

## 七、输入材料清单

| 类别 | 材料 | 必须/可选 |
|------|------|-----------|
| **学生文稿** | 完整英文稿件（.docx/.tex/.md） | ✅ 必须 |
| **竞赛信息** | 竞赛名称、类别、截止日期 | ✅ 必须 |
| **评分标准** | 官方 rubric（PDF/文本） | ✅ 必须 |
| **字数要求** | 最低/最高字数 | 🟡 建议 |

| **往年范文** | 往届获奖作品 | 🔵 可选 |
| **教师经验** | 教师辅导心得/经验总结（.md） | 🔵 可选 |

## 八、校准引擎与教师经验注入

校准引擎是 AcademicReviewer 的 **Phase 3** 核心模块，它不属于 5 Agent 评审管道，而是独立的**配置优化系统**。引擎通过分析获奖/失败文章的统计差异（Cohen's d 效应量），自动生成配置文件更新建议。在此基础上，系统新增**教师经验注入**通道，允许教师将多年辅导经验以文档形式纳入校准流程，实现"数据驱动 + 教师经验"的双重校准。

### 八点一、校准引擎概述

```
                    ┌─────────────────────────┐
                    │     Feature Extractor    │
                    │   21维特征统一提取        │
                    │  (winners vs losers)     │
                    └───────────┬─────────────┘
                                │
                    ┌───────────▼─────────────┐
                    │     Cohen's d Engine     │
                    │  效应量计算 + 显著性检验   │
                    │  d≥0.8→大效应 d<0.2→忽略  │
                    └───────────┬─────────────┘
                                │
        ┌───────────────────────┼───────────────────────┐
        │                       │                       │
  ┌─────▼─────┐          ┌─────▼─────┐          ┌─────▼─────┐
  │ 统计差异   │          │ 教师经验   │          │ 外部验证   │
  │ (纯数据)   │          │ (文档注入) │          │ (交叉验证) │
  │ 零 LLM    │          │ 选择性 LLM │          │ 零 LLM    │
  └─────┬─────┘          └─────┬─────┘          └─────┬─────┘
        └───────────────────────┼───────────────────────┘
                                │
                    ┌───────────▼─────────────┐
                    │     Diff Generator      │
                    │  配置差异 → 变更建议      │
                    │  含经验-统计交叉验证      │
                    └───────────┬─────────────┘
                                │
                    ┌───────────▼─────────────┐
                    │     Report Generator    │
                    │  校准报告 → 5 Agent 配置  │
                    └─────────────────────────┘
```

校准引擎的核心价值在于**闭环反馈**——每次比赛结束后，教师可以收集本届获奖和失败文章，运行校准引擎，系统自动分析出哪些特征区分了高分和低分，并给出配置文件调整建议。这些调整会在下一轮评审中自动生效，实现"越用越准"的效果。

### 八点二、教师经验注入：设计动机

**问题**：纯统计校准能告诉你"获奖文章有什么特征"，但无法解释"为什么这些特征重要"。

**场景**：
- "我带过 3 届 ISEF 获奖学生，我发现评审特别喜欢看到学生对实验局限性的反思"
- "很多学生在 HiMCM 中忽略了模型假设的检验，这是最大的扣分点"
- "FBLA 的评审往往在 Introduction 前 3 句就决定了印象分"

这些教师经验是解读统计数据的关键上下文。例如，统计发现 `thesis_clarity` 的 Cohen's d = 1.2（大效应），教师的经验可以补充说明"评审希望看到一段式简洁开篇，而非冗长背景铺陈"——这使得配置变更建议更精准、更具可操作性。

### 八点三、三模式自适应解析

为了**零门槛**接受教师的经验文档，系统支持三种文档格式，自动检测并选择合适的解析方式：

| 模式 | 格式要求 | 解析方式 | LLM 成本 | 适合场景 |
|------|----------|----------|----------|----------|
| **Mode 3** 完全结构化 | `### [pattern] feature_name` 格式，每条后跟描述 | 正则精确解析 | **零 LLM** | 教师愿意严格按模板编写 |
| **Mode 2** 半结构化 | 使用章节标题分节（如 `## 获奖经验`），节内自由撰写 | 章节路由 + LLM 逐节抽取 | **1 次 LLM** | 教师愿意按章节组织内容 |
| **Mode 1** 纯自由格式 | 无任何格式要求，纯随笔 | LLM 全文语义解析 | **1 次 LLM** | 最低门槛，随手写心得 |

**模式自动检测逻辑**：

```
detect_mode(text):
    1. if "### [" in text         → return 3  （优先级最高）
    2. elif 已知章节标题 in text   → return 2
    3. else                       → return 1
```

**统一入口** `unify_parse(file_path, llm=None)` 对上层完全透明：
- 检测到 Mode 3 → 调用 `parse_expert_document()`（正则，零 LLM）
- 检测到 Mode 2/1 → 调用 `parse_freeform_expert_doc()`（需要 LLM 实例）
- Mode 1/2 无 LLM 时抛出明确错误

### 八点四、Mode 2 章节标题体系

为 Mode 2 教师提供 5 大类共 20+ 个中文章节标题，系统自动识别归类：

| 类别 | 标注类型 | 支持的标题（任选其一） |
|------|----------|----------------------|
| 获奖经验 | `pattern` | 获奖经验、获奖规律、成功案例总结、高分特征、高频特征、晋级规律、获奖论文的共同特征、制胜关键、历届获奖作品特点、我观察到的得分规律、学生获奖的共性 |
| 常见问题 | `pitfall` | 常见问题、常见错误、失分原因、扣分点、评审中常见的问题、学生常犯错误、需要避免的问题、失败案例分析、容易忽略的点、我踩过的坑、常见误区、低级失误 |
| 高分信号 | `signal` | 高分信号、加分项、评委偏好、评委在意的点、能打动评委的细节、评审看重的、出彩的技巧、区分度高的特征、脱颖而出的关键、拿分的细节 |
| 教学方法 | `strategy` | 教学方法、辅导策略、有效的训练方法、我常用的教学技巧、提升途径、备考策略、写作训练方法 |
| 评分标准解读 | `rubric` | 评分标准解读、评分细则分析、我对评分表的理解、如何理解评审标准、得分要点分析、各维度的评分侧重点 |

### 八点五、LLM 调用策略与安全设计

教师经验注入的核心挑战是**准确性与灵活性之间的平衡**。系统采用以下策略：

**LLM 只负责文本拆分，不负责特征映射**：

```
LLM 任务边界：
  ✅ 输入：一段教师自由文本
  ✅ 输出：[{title, description, annotation_type}]
  ❌ 不做：猜测这条经验对应配置文件的哪个具体特征

特征映射（本地确定性）：
  1. 将 LLM 返回的 title + description 与本地 FEATURE_KEYWORDS 字典匹配
  2. 关键词匹配 → 确定具体特征维度（如 "thesis_clarity"）
  3. 匹配不上的 → 保留原始描述，标注为"未映射"
```

**安全性保障**：即使 LLM 产生幻觉（编造特征名、错误归类），最终的特征映射仍然走本地关键词匹配，**零幻觉风险**。

**成本控制**：
- Mode 3：零 LLM 调用，零成本
- Mode 2/1：每个文档 **1 次 LLM 调用**，成本 ≈ 一次评审的 1/3
- 无 API Key 时自动拒绝 Mode 1/2，并给出明确提示
- Mode 2 在 LLM 失败时回退到启发式提取（按首行拆分）

### 八点六、与 5 Agent 管道的集成

教师经验通过校准引擎间接影响 5 Agent 评审管道：

```
教师经验文档
    │
    ▼
[三模式解析] → ExpertInsights (知识点集合)
    │
    ▼
[校准引擎] → 统计差异 + 教师经验 → 交叉验证
    │
    ▼
[校准报告] → 配置变更建议（含"教师经验验证通过"标注）
    │
    ▼
[配置文件] → rubric_calibrated.json / structure_schema_calibrated.json 等
    │
    ▼
[5 Agent 管道] → 下一轮评审自动使用更新后的配置
```

**集成路径**：
1. **CLI**：`python cli/calibrate.py --expert-docs doc1.md doc2.md`
2. **Gradio**：校准标签页新增"教师经验文档"上传组件
3. **engine.py**：自动检测 expert_doc_paths，自动创建 LLM 适配器

**报告新增章节**：
- 教师经验摘要（按类型：pattern/pitfall/signal/strategy/rubric）
- 经验与统计交叉验证（"统计验证通过" / "需进一步观察"）
- 融合建议（综合统计 + 经验）

## 十、开发注意事项

1. **A4 的铁律是学生主权的最后防线**：如果 prompt 里不写死改写权限边界，A4 会在你没有察觉的情况下替学生改风格、改用词、改句式，最终稿件不再是学生自己的作品
2. **A5 尽量多走规则引擎少走 LLM**：引用配对是确定性逻辑（cite ↔ ref），用正则匹配比 LLM 更快、更准、更便宜。LLM 仅在最终综合判断时调用
3. **A1 的 rubric 解析只做一次**：结果缓存，不要每篇审查都重新解析
4. **A3 和 A4 是 L2 扩展载体**：新增赛事类型时，主要开发工作是 A3 的审查逻辑 + A4 的风格模板 + 对应的配置文件。9 种竞赛类型中，仅数模型和商科型需新增独立 L2 配置
5. **配置文件是 Calibration Engine 和审查管线的唯一数据契约**：审查管线不关心配置怎么来的，Calibration Engine 不关心配置被谁消费

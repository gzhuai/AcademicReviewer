# 学术竞赛文稿审查智能体 —— 完整设计方案 v1.0

---

## 一、核心定位与设计原则

### 1.1 核心定位

这个系统的本质是一个 **"AI 学术导师 + 质量门禁"**，而非代写工具。它的核心价值在于：

> 在不替代学生思考的前提下，系统性地发现文稿中的问题，并以**可操作的反馈**指导学生自主改进。

### 1.2 设计原则（五项铁律）

| 原则 | 说明 |
|------|------|
| **学生主权** | AI 只做批注和建议，绝不直接改写正文，确保作品仍为学生原创 |
| **分层审查** | 从宏观（论点结构）到微观（语法拼写），逐层把关，避免挂一漏万 |
| **竞赛对齐** | 每次审查都与具体竞赛的评分标准（rubric）严格对标 |
| **可解释性** | 每个建议必须附带理由和评分维度，让学生"知其所以然" |
| **迭代闭环** | 支持多轮提交→审查→修改→再审，跟踪质量收敛趋势 |

## 二、系统架构：Multi-Agent 协作体系

采用 **编排器 + 专家智能体池（Orchestrator + Agent Pool）** 架构：

```
┌──────────────────────────────────────────────────────────┐
│                     Orchestrator                         │
│              (任务调度 / 结果聚合 / 质量报告)               │
└──────┬───────┬───────┬───────┬───────┬───────┬──────────┘
       │       │       │       │       │       │
   ┌───▼──┐┌──▼───┐┌──▼───┐┌──▼───┐┌──▼───┐┌──▼──────┐
   │Rubric││Logic ││Argue ││Style ││Citation││Original│
   │Parser││&Flow ││&Evid ││&Voice││Checker││  Check  │
   └──────┘└──────┘└──────┘└──────┘└───────┘└─────────┘
```

### 2.1 七个专业 Agent 的职责

#### Agent 1: Rubric Parser（评分标准解析器）
- **输入**：竞赛官方 rubric（PDF/图片/文本）
- **输出**：结构化的评分维度树 + 权重 + 关键词
- **能力**：将模糊的评分描述（如 "demonstrates critical thinking"）转化为可检查的具体指标

#### Agent 2: Structure & Logic（结构与逻辑审查）
- 检查论文整体骨架：引言→文献综述→方法→论证→结论的完整性
- 检测段落间的逻辑衔接（是否存在跳跃、重复、缺失环节）
- 评估 thesis statement 的清晰度和全文一致性

#### Agent 3: Argument & Evidence（论点与证据审查）
- 核查每个 claim 是否有充分 evidence 支撑
- 识别逻辑谬误（因果倒置、稻草人、以偏概全等）
- 评估证据来源的权威性和时效性

#### Agent 4: Academic Style & Voice（学术风格与语气）
- 检查学术写作规范（避免口语化、情绪化表达）
- 术语使用的一致性
- 被动语态/主动语态的合理分布
- 是否符合目标竞赛的预期风格（如 ISEF 偏科研、John Locke 偏思辨）

#### Agent 5: Citation & Format（引用与格式审查）
- 检查所有引用是否完整（有 cite 必有 reference，反之亦然）
- 验证引用格式（APA / MLA / Chicago 等）的一致性
- 检测可疑的 AI 生成引用（虚假文献检测）

#### Agent 6: Originality Guardian（原创性守护）
- 检测潜在抄袭/过度借鉴段落（与内部历史稿件库比对）
- 识别大段 AI 生成痕迹（语气突变、模板化结构）
- 确保学生个人 voice 得以保留

#### Agent 7: Language Polishing（语言精修）—— 唯一允许改写的 Agent
- 语法、拼写、标点修正
- 中式英语 → 地道表达的建议
- 长难句拆分建议
- **严格以批注模式工作，不直接改动原文**

## 三、工作流设计：五阶段管道

### 阶段流程图

```
输入阶段          审查阶段                        反馈阶段        迭代阶段
┌──────┐    ┌─────────────────────────┐    ┌──────────┐    ┌──────────┐
│ Stage1│───▶│ Stage2: Multi-Agent Review│───▶│ Stage3:  │───▶│ Stage4:  │
│ Intake│    │  Rubric→Structure→Logic    │    │ Feedback │    │ Revision │
│       │    │  →Evidence→Style→Citation  │    │ Report   │    │ Support  │
└──────┘    └─────────────────────────┘    └──────────┘    └──────────┘
                                                               │
                                                          ┌────▼─────┐
                                                          │ Stage5:  │
                                                          │ Final QC │
                                                          │ & Export │
                                                          └──────────┘
```

### Stage 1: 材料摄入与预处理（Intake）

**输入材料清单：**

| 类别 | 材料 | 必须/可选 | 说明 |
|------|------|-----------|------|
| **学生文稿** | 完整英文稿件（.docx / .tex / .md） | ✅ 必须 | 支持带格式文档 |
| **竞赛信息** | 竞赛名称、类别、提交截止日期 | ✅ 必须 | 用于加载对应 rubric 模板 |
| **评分标准** | 官方 rubric / judging criteria | ✅ 必须 | 最好是 PDF 原文，文本也可 |
| **字数要求** | 最低/最高字数限制 | 🟡 建议 | 超字数或字数不足的检测 |
| **格式要求** | 字体、行距、页边距等 | 🟡 建议 | 格式合规自动检查 |
| **往年范文** | 该竞赛往届获奖作品 | 🔵 可选 | 用于风格对标和质量基准 |
| **学生背景** | 年级、英语水平、过往成绩 | 🔵 可选 | 个性化反馈深度 |
| **历史版本** | 该学生此文稿的前序版本 | 🔵 可选 | 用于追踪改进轨迹 |
| **导师备注** | 辅导老师对该文稿的初步判断 | 🔵 可选 | 辅助 AI 聚焦关键问题 |

**预处理操作：**
- 将 .docx/.pdf 转换为结构化 Markdown（保留标题层级）
- 自动分段、分句、分段落类型标注（intro/body/conclusion）
- 统计字数、段落数、引用数等基础指标

### Stage 2: 多 Agent 并行审查

**核心思路：先粗后细，先整体后局部。**

```
Orchestrator 调度顺序:

Round 1（宏观）- 并行执行:
  ├── Agent 1: Rubric Parser → 生成评分检查清单
  ├── Agent 2: Structure & Logic → 结构完整性报告
  └── Agent 3: Argument & Evidence → 论证力评估

Round 2（微观）- 基于 Round 1 结果执行:
  ├── Agent 4: Academic Style → 风格一致性检查
  ├── Agent 5: Citation Check → 引用完整性检查
  └── Agent 7: Language Polish → 语言问题标注

Round 3（综合）:
  └── Agent 6: Originality Guardian → 原创性审查（需要前两轮结果）
```

**关键设计决策：分离审查粒度**

- **宏观看结构**：就算语法完美，论点站不住脚 = 不及格
- **微观查细节**：结构再好，引文格式全错 = 扣分
- **先别急着改语法**，很多学生会因为改语法而产生"已经改好了"的错觉，忽略了更大的结构性问题

### Stage 3: 反馈报告生成

反馈报告不是简单的错误列表，而是分层级的：

```markdown
# 学术竞赛文稿审查报告

## 📊 总体评分预估
| 维度 | 评分 | 权重 | 加权 |
|------|------|------|------|
| 论点与论证 | 6.5/10 | 30% | 1.95 |
| 结构与逻辑 | 7.0/10 | 25% | 1.75 |
| 学术规范 | 8.0/10 | 20% | 1.60 |
| 语言表达 | 7.5/10 | 15% | 1.13 |
| 原创性   | 9.0/10 | 10% | 0.90 |
| **加权总分** | | | **7.33/10** |

## 🔴 Critical Issues（必须修改）
- [ ] Thesis statement 在第2段之后出现偏离...
- [ ] 第4段的 claim 缺少 citation 支撑...

## 🟡 Major Suggestions（强烈建议）
- [ ] 方法论部分缺少对样本量的说明...
- [ ] 讨论部分未回应引言中提出的核心问题...

## 🔵 Minor Improvements（锦上添花）
- [ ] 第3段可考虑增加一个过渡句...
- [ ] 关键词 "sustainability" 出现了12次，建议用同义词替换部分...

## 💡 Competition-Specific Tips
- ISEF 评委通常对实验可复现性特别关注，建议补充...
```

### Stage 4: 学生修改辅助

这个阶段提供的是**不代写的辅助工具**：
- 针对每条 Critical Issue 提供引导性提问（例如："你的实验中，控制组和实验组的关键区别是什么？请尝试在方法部分补充"）
- 提供同义表达库（不直接改句子，而是给 3-5 个可选的表达方向）
- 修改后自动触发增量审查（只审查修改过的段落 + 受影响的上下游段落）

### Stage 5: 终审与导出

- 最终合规检查（字数、引用格式、文件命名等硬性要求）
- 自动生成提交清单（Checklist）
- 导出带批注的 .docx 或 PDF（批注为评论模式，便于人工导师最终把关）

## 四、技术实现建议

### 4.1 推荐的 Prompt 工程策略

每个 Agent 使用**结构化 Prompt 模板**，至少包含：

```
[Role] 你是XXX领域的竞赛评委，拥有15年评审经验
[Context] 这是参加XXX竞赛的高中生论文，竞赛rubric如下...
[Task] 请逐段审查本文的结构逻辑，重点关注...
[Output Format] 请严格按以下JSON格式输出...
[Constraints] 不要重写学生的句子，只指出问题和改进方向
[Examples] 好的反馈 vs 差的反馈示例...
```

### 4.2 与现有 LLM 的适配

鉴于在中国大陆运营，建议考虑：
- **模型层抽象**：同时支持 OpenAI / Claude / 国产模型（DeepSeek、Qwen等），通过适配层切换
- **长文本处理**：竞赛论文可能 3000-10000 词，需要分段处理 + 跨段上下文窗口
- **引用校验**：结合 Google Scholar / Semantic Scholar API 做引用真实性校验

### 4.3 关键功能模块

| 模块 | 功能 | 优先级 |
|------|------|--------|
| Rubric 模板库 | 预置 20+ 常见竞赛的评分标准模板 | 🔴 高 |
| 版本对比 | 多轮修改的 diff 视图 | 🔴 高 |
| 导师面板 | 人工导师可覆盖/调整 AI 建议 | 🟡 中 |
| 批量处理 | 一次上传多篇，批量审查 | 🟡 中 |
| 学习仪表盘 | 追踪学生写作能力的成长曲线 | 🔵 低 |

## 五、风险与边界

| 风险 | 应对策略 |
|------|----------|
| **学生过度依赖 AI** | 系统只做批注不改写；导师最终确认；保留完整的审查日志 |
| **AI 产生幻觉引用** | 所有引用建议必须经过 API 校验；虚假引用标注为 Critical Issue |
| **不同竞赛要求冲突** | 为每个竞赛维护独立 rubric 模板，避免通用化建议 |
| **学生个人 voice 丧失** | Agent 6 专门守护原创性；任何风格建议标注为可选 |

## 六、实施路线图

```
Phase 1（MVP，2-4周）:
├── 单 Agent 审查（Language + Structure）
├── 手动粘贴 Rubric
├── 文本格式输入/输出
└── 目标：跑通核心审查流程

Phase 2（增强，1-2个月）:
├── 完整 7-Agent 体系
├── Rubric 模板库
├── 多轮迭代支持
└── 目标：覆盖 5+ 竞赛类型

Phase 3（产品化，2-3个月）:
├── 导师管理面板
├── 批量处理
├── 学习仪表盘
└── 目标：辅导机构内部量产使用
```

一、设计哲学

原则：教练不替代学生判断。教练告诉你缺了什么、为什么是问题、然后该怎么改。教练不会替你写，但会帮你把思考路径走通。

1.1 三项铁律

铁律说明反例
缺→问→改先说缺了什么，再说为什么是问题，最后说该怎么改"结构不完整" → 应该说"缺少 Methods 章节，没有它读者无法复现实验。你应该在第2段和第3段之间插入一个 Methods 段，包含实验设计、材料和统计方法。"
不硬编码Prompt 中不写死任何竞赛专属的章节名、检查项、JSON 枚举值sections_found 硬编码为 ["Introduction / Hook", "Definition & Framework"...] → 应该写"从提供的 structure schema 中读取 sections"
类型切换竞赛专属检查通过 agent 代码注入 type hint，LLM 根据 hint 决定检查什么对 ISEF 论文要求 counterargument → 应该识别 type=research，改查 limitations 讨论

1.2 v1.6 → v1.7 的核心变更

变更v1.6 及之前v1.7
sections_found硬编码为 discursive 结构"从提供的 structure schema 中读取"
thesis_coaching仅针对 discursive thesis改为 main_argument_coaching，科研=hypothesis 验证，思辨=thesis 稳定性
counterargument_analysis强制所有竞赛检查反方改为 validation_point，根据类型切换：counterargument / limitation_review / sensitivity_check / alternative_comparison
fallacy 类型仅 discursive 常用逻辑谬误新增科研/金融专用：correlation_not_causation, survivorship_bias, hindsight_bias
quantitative_checkv2.0 中被误删回归，科研/数模需要检查 P值/效应量/灵敏度
A3 agent 代码不传竞赛类型提示注入 type hint 告诉 LLM 该检查什么、不该检查什么

二、双层架构

2.1 架构图

┌──────────────────────────────────────────┐
│            Prompt .txt 文件               │
│  ┌────────────────────────────────────┐  │

│  └────────────────────────────────────┘  │
│                                          │
│  ┌────────────────────────────────────┐  │
│  │   竞赛专用层 (Agent 代码注入)        │  │
│  │  · structure schema 全文           │  │
│  │  · evidence patterns 配置          │  │

2.2 数据流

competition_registry.json → type (如 "research")
  → A2: 加载 structure_schemas/research.json → 注入 user_message
  → A3: 加载 evidence_patterns/research.json → 注入 user_message + type hint
  → LLM: 读 system prompt (通用层) + user message (竞赛专用) → 生成反馈

2.3 为什么不用变量替换

Prompt 文件使用自然语言指令而非 {PLACEHOLDER} 模板变量。

  • 自然语言指令（如"从提供的 structure schema 中读取"）让 LLM 自己理解上下文并适应
  • 变量替换（如 {SECTIONS_LIST}）把 schema 按字符拼接进 prompt，容易超 token 限制，且 LLM 可能不理解列表格式
  • 当前方式：schema 以 JSON 形式放在 user message 中，LLM 原生理解 JSON 结构

三、教练型反馈原则

3.1 "缺→问→改" 三步法

步含义示例
缺明确说出缺了什么"你的 Discussion 章节缺少对 effect size 的报告"
问解释为什么这是问题"P值只能告诉你差异是否显著，但不能告诉你差异有多大。评委需要看到效应量来判断你的发现是否有实际意义"
改给出可以直接使用的改写方案"在 Results 最后加上：Cohen's d = 0.78 (95% CI: 0.52–1.04), indicating a large effect. 你可以把这句话直接放在 p<0.01 后面"

3.2 语言风格约定

使用避免
"你要把 thesis 往更稳的方向收""建议 thesis 不要过于绝对"
"这一段只保留一个任务：解释 death anxiety 为什么让遗体处理变得 emotionally charged""这一段内容有点多，可以考虑拆分"
"查不清的 source 先删，不要赌""建议核实引用来源的真实性"

3.3 正向反馈先行

所有 Agent 输出中，positive_points 必须放在 key_issues 前面。"先肯定做得好的，再说需要改的"——这和人类教练的工作方式一致。

四、A2 StructureLogic 规范

4.1 通用教练层（写死在 prompt 文件）

你是学术竞赛文稿的结构与逻辑审查教练。
反馈风格：先说缺了什么，再说问题是什么，最后告诉学生该怎么改。

【通用审查原则】
1. 检查给定的 structure schema 中列出的所有必需章节
2. 每指出一个结构问题，给出该段落的功能定义
3. 逻辑衔接问题给一句可以直接用的过渡句
4. 检查段落间的内容重复

4.2 竞赛专用层（Agent 代码注入）

注入内容来源说明
竞赛类型提示competition_type 参数如 "research", "discursive", "finance"
结构 schema 全文configs/structure_schemas/{type}.jsonrequired_sections, expected_ratio, logic_checks, high_severity_violations

4.3 输出字段

字段类型说明
structure_scorefloat0-10
sections_foundlist[str]从 schema 中匹配到的章节名。必须用 schema 中的名称，不硬编码
missing_sectionslist[str]缺失的必需章节
section_issueslist[dict]含 current_problem / should_only_do / what_to_move / severity / hint
logic_issueslist[dict]含 transition_suggestion（可直接使用的过渡句）
main_argument_coachingdictcurrent_statement, vulnerability, stronger_version。科研型关注 hypothesis 可验证性，思辨型关注 thesis 反驳抗性
duplication_reportlist[dict]含 repeated_content 和 fix 方案

五、A3 ArgumentEvidence 规范

5.1 通用教练层

反馈风格：先说缺了什么证据/逻辑，再说问题是什么，最后告诉学生该怎么改。

【通用审查原则】
1. 每指出一个缺陷，给可操作的具体改写方案或缺失的推理环节
2. Claim-Evidence 匹配：证据类型取决于竞赛——学术引用/实验数据/财务数字/公式推导/案例分析
3. 逻辑谬误：给出修正后的推理形态

【竞赛专用检查】
思辨型：counterargument 质量、rebuttal 强度
科研型：hypothesis 可验证性、P值/效应量、局限性讨论
数模型：模型假设合理性、灵敏度分析
金融型：估值模型假设、数据可溯性
商科型：框架应用正确性、ROI 计算

5.2 竞赛专用层（Agent 代码注入）

# argument_evidence.py _build_user_message()
type_hints = {
    "research": "科研型论文。检查 hypothesis 可验证性、p值/效应量、控制组、统计检验、局限性讨论。不要检查 counterargument/rebuttal。",
    "discursive": "思辨型议论文。检查 counterargument 质量、rebuttal 强度、逻辑链完整性、哲学引用深度。",
    "math_modeling": "数学建模论文。检查模型假设合理性、求解可追溯、灵敏度分析。不要检查 counterargument/rebuttal。",
    ...
}
hint = type_hints.get(competition_type, "")
# 注入到 user_message 中: f"竞赛专项提示：{hint}"

六、A4 LanguageStyle 规范

6.1 Iron Law（改写权限边界）

A4 是唯一允许改写的 Agent。边界必须绝对清晰：

可改写 ✅不可改写 ❌（仅批注）
拼写错误句式选择
语法错误（主谓一致、时态、冠词、介词、单复数）语态分布调整
标点修正用词偏好替换
Chinglish 语法修正（不影响原意）段落结构/节奏调整
—风格判断改写

6.2 教练风格

对于拼写错误，不要只说"有拼写错误"——把错误原文和纠正后的文本并列列出。
反馈口吻：教练型。直接说"这里有问题，改成这样"，不说"建议考虑修改"。
输出分两个区：[REWRITE] 可改写的，[SUGGEST] 仅批注的。

七、A5 AcademicIntegrity 规范

7.1 两条独立路径

  • 5a Citation Check：规则引擎优先（正则匹配），LLM 仅做最终综合判断。检查 cite↔ref 配对、格式一致性、疑似幻觉引用
  • 5b Originality Check：ChromaDB 向量比对 + AI 生成痕迹检测 + 个人 voice 保留度评估

7.2 教练风格嵌入

A5 的 prompt 保持现有格式，但在 citation_report 中增加 coach_note 字段：

"coach_note": "查不清的 source 先删，不要赌。正文的 in-text citation 必须能找到 reference，每条 reference 的标题/作者/年份/期刊/页码都必须完整。"

八、validation_point 类型表

核心设计：A3 输出中的 validation_point.type 字段根据竞赛类型动态切换语义，确保 LLM 不会对 ISEF 论文要求 counterargument。

竞赛类型validation_point.type检查语义是否需要 counterargument
思辨型 (discursive)counterargument_analysis反方是否被推到最强？rebuttal 是否把反方接回来？✅ 核心检查项
科研型 (research)limitation_review是否讨论了局限性？样本量、混杂变量、外部有效性？❌ 不适用
科研进阶型limitation_review同科研型 + 理论贡献度和方法原创性❌ 不适用
数模型sensitivity_check参数变化后模型是否稳定？假设是否被检验？❌ 不适用
金融型null不适用——金融报告的核心是估值模型和数据溯源❌ 不适用
商科型alternative_comparison是否比较了其他可行方案？为什么选方案A不选方案B/C？❌ 用 alternative comparison 替代
社科型null核心是调研方法论和案例深度❌ 不适用
历史型null核心是一手史料和史学视角❌ 不适用

8.1 validation_point JSON schema

{
  "type": "counterargument_analysis",  // 从表中选取
  "current_strength": "weak|moderate|strong",
  "coach_guidance": "教练指导：缺什么、为什么重要、怎么改",
  "word_count_suggestion": "如果适用，建议该部分的字数范围"
}

九、字段命名与向后兼容

9.1 重命名对照

旧名称 (v1.6)新名称 (v1.7)原因
thesis_coachingmain_argument_coaching科研型没有 thesis，有 hypothesis；数模型和商科型是核心论证而非 thesis
counterargument_analysisvalidation_point科研型不检查 counterargument，需要类型切换
thesis_coaching.current_thesismain_argument_coaching.current_statement语义泛化：可能是 thesis / hypothesis / investment thesis / recommendation

9.2 消费者兼容

所有下游消费者（annotation_builder、report 生成器、Gradio UI）必须以兼容方式读取新旧字段：

tc = structure.get("main_argument_coaching") or structure.get("thesis_coaching", {})
vp = argument.get("validation_point") or argument.get("counterargument_analysis", {})

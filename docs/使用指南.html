1. 系统概述

AcademicReviewer 是一个基于多 Agent 协作的学术竞赛论文智能评审系统，专为辅导高中生参加国际学术竞赛的教师团队设计。

🎯 核心价值

解决教师团队在辅导多名学生时评审质量不一致的核心痛点。系统通过 5 个专用 Agent 两轮并行协作，确保每篇稿件的评审标准高度统一。

系统架构

组件说明
5 个评审 AgentA1 评分标准解析、A2 结构与逻辑、A3 论证与证据、A4 语言与风格、A5 学术诚信
2 轮并行调度Round 1: A1+A2+A3 并行 → Round 2: A4+A5 并行
校准引擎纯统计分析（无 LLM 调用），分析获奖/失败文章差距，生成配置优化建议
4 模型适配DeepSeek / GPT / Gemini / GLM，一键切换
11 项竞赛ISEF、丘成桐、John Locke、HiMCM、SIC 等，不同竞赛自动加载专属评审配置

2.1 Windows 一键安装（双击即用）

1

脚本会自动完成以下操作：

▲ 图3：提交评审标签页

▲ 图6：校准引擎标签页

  1. 自动提取每篇文章的 21 项文本特征
  2. 计算 Cohen's d 效应量（识别关键差异特征）
  3. 生成配置优化建议
  4. 输出 Markdown 校准报告

操作步骤：

  7. 点击 "运行校准"
💡 提示：校准引擎是纯统计分析，不调用任何 LLM，完全免费运行。建议每积累 20+ 篇文章后运行一次。

5. 校准引擎

校准引擎是系统的质量持续改进工具，通过统计分析获奖文章与失败文章的差异，自动生成配置优化建议。

🧪 校准引擎特点

  • 零 LLM 调用：纯规则引擎 + 统计分析，完全免费
  • 21 项特征提取：句式参数、引用密度、被动语态比例、词汇多样性、逻辑标记密度等
  • Cohen's d 效应量：识别获奖文章与失败文章之间的关键差异特征（|d|≥0.8=关键差异）
  • 交叉验证：支持引入外部获奖文章进行特征验证
  • 配置变更建议：自动将统计发现映射到配置参数调整建议

5.1 准备工作

在运行校准之前，需要准备以下文件结构：

data/
└── calibration/
    ├── winners/        ← 获奖文章文件夹 (.txt/.md)
    │   ├── paper1.txt
    │   └── paper2.txt
    ├── losers/         ← 失败文章文件夹 (.txt/.md)
    │   ├── paper3.txt
    │   └── paper4.txt
    └── external/       ← 外部获奖文章（可选，用于交叉验证）
        └── ref_paper1.txt

5.2 CLI 命令行方式

python cli/calibrate.py \
  --competition ISEF \
  --type research \
  --winners data/calibration/winners \
  --losers data/calibration/losers \
  --external data/calibration/external \
  --expert-docs data/expert_insights/ISEF_teacher_experience.md \
  --output data/calibration/report.md

参数说明：

参数必填说明

--type✅竞赛类型（如 research、discursive）
--winners✅获奖文章目录

5.4 报告解读

校准报告包含以下部分：

章节内容
效应量排序表按 |d| 降序排列，标注关键差异/中效应/小效应/无差异
交叉验证表与外部获奖文章的对比验证
配置变更建议具体的配置参数调整方案
致命缺陷检测失败文章中普遍缺失的二元特征（如无 p 值、无对照组）
人工复核清单需要教师确认的关键发现

5.5 教师经验注入

校准引擎支持将教师多年的辅导经验以文档形式注入校准流程，使配置优化建议不仅基于统计数据，还融合了教师的主观经验判断。

🎓 为什么需要教师经验注入？

纯粹的统计校准只能告诉你"获奖文章有什么特征"，但无法解释"为什么这些特征重要"。教师多年的辅导经验——例如评委偏好、常见失分原因、教学心得——恰恰是解读统计数据的关键上下文。将这些经验纳入校准，报告中的"配置变更建议"将更精准、更有可操作性。

5.5.1 三模式自适应解析

系统支持三种文档格式，教师可以根据自己的习惯自由选择。引擎会自动检测文档格式并选择合适的解析方式：

模式格式要求解析方式适合场景
Mode 3
完全结构化		 使用 ### [pattern] 特征名 格式编写，每个条目后跟描述 正则精确解析，零 LLM 调用 教师愿意严格按模板编写时，精度最高
Mode 2
半结构化		 使用系统提供的章节标题（见下方标题体系）将文档分成若干小节，每节自由撰写 章节路由 + LLM 逐节抽取，特征映射为本地关键词匹配 教师愿意按章节组织内容，但每节内容自由发挥
Mode 1
纯自由格式		 没有任何格式要求，纯粹的随笔、心得、总结 LLM 全文语义解析 + 本地关键词特征映射 教师只想随手写写心得，最低门槛
💡 安全性设计：Mode 1 和 Mode 2 中，LLM 只负责将文本拆分为知识点条目（标题 + 描述 + 类型），而特征映射始终走本地关键词匹配。这意味着即使 LLM 出现幻觉，也不会错误地将经验映射到不相关的特征上，保证了校准建议的准确性。

5.5.2 Mode 2 章节标题体系（20+ 标题）

以下是系统支持的章节标题，分为 5 大类。教师选择任意一个标题来组织自己的经验，系统将自动识别并归类：

类别标注类型支持的章节标题（任选其一即可）
获奖经验 pattern 获奖经验、获奖规律、成功案例总结、高分特征、高频特征、晋级规律、获奖论文的共同特征、制胜关键、历届获奖作品特点、我观察到的得分规律、学生获奖的共性

6. 团队协作：数据集中汇总

系统支持「本地运行 + 中央汇总」架构，让所有同事的使用数据和校准结果自动归集到管理员处。同事完全不需要接触 Git 或任何复杂操作。

▲ 团队协作架构：同事本地运行 → 自动上报 → 中央服务器

6.1 工作原理

同事每次提交评审或运行校准后，系统会自动上报一份摘要数据（评审分数、校准结果、竞赛类型等）到你部署的中央服务器。不会上传原始论文内容，只上传结构化的统计摘要。

工作流程：

  1. 你（管理员）部署一台中央服务器（可以是云服务器或有公网 IP 的机器）
  2. 同事在本地安装系统时，在 .env 中填入你的服务器地址和自己的名字
  3. 同事每次使用系统的评审或校准功能，数据自动、静默上报到你的服务器
  4. 你打开浏览器，进入「管理看板」即可查看所有汇总数据

6.2 管理员：部署中央服务器

1

准备服务器

找一台云服务器（阿里云、腾讯云等）或你办公室里有公网 IP 的机器，按正常流程安装系统（参考 快速开始）。

2

编辑服务器上的 .env

和普通安装一样填入 API Key，但额外注意两点：

# 让服务器对外可访问（监听所有网络接口）
HOST=0.0.0.0

# 中央服务器本身不上报，所以留空
SYNC_SERVER_URL=
3

启动并记下地址

启动系统后，你就有了两个地址：

地址用途示例
后端 API同事配置时需要http://你的服务器IP:8000
Gradio 前端你查看管理看板http://你的服务器IP:7860

把后端 API 地址发给同事。

⚠️ 安全提示：如果服务器有防火墙（云服务器通常有安全组），需要放行 8000 和 7860 端口，否则同事无法上报数据。

6.3 同事：配置自己的名字和服务器地址

1

打开 .env 文件

同事在安装系统时，完成 API Key 配置后，在 .env 文件末尾找到（或添加）以下两行：

# 中央服务器地址（管理员提供给你）
SYNC_SERVER_URL=http://管理员的服务器IP:8000

# 你的名字（会显示在管理看板上，用什么名字都行）
INSTANCE_NAME=刘老师

▲ 同事在 .env 中配置 SYNC_SERVER_URL 和 INSTANCE_NAME

INSTANCE_NAME 就是你的名字：可以写真名（如「刘老师」）、电脑名、编号——任意你喜欢的标签。这个名字会显示在管理员看板上，用于区分不同同事的使用数据。如果不填，系统会自动使用电脑的主机名。
2

正常使用即可

配置完成后，无需做任何额外操作。每次提交评审或运行校准，数据会自动上报。

同事可以在「系统状态」标签页查看自己的同步配置：

  • 如果显示「中央服务器: http://xxx」且「本机标识: 刘老师」，说明配置已生效
  • 如果显示「中央服务器: 未配置」，说明 SYNC_SERVER_URL 为空
3

同步管理员调教后的最新配置（重点！）

当管理员根据校准结果调优了评分标准后，同事可以从中央服务器拉取最新配置：

  1. 打开浏览器进入系统 → 「系统状态」 标签页
  2. 展开底部的 「从中央服务器同步最新配置」 折叠面板
  3. 点击 「预览变更」，查看哪些配置文件会被新增、更新或删除
  4. 确认无误后，点击 「确认同步」
  5. 系统会自动备份旧配置到 data/configs_backup_时间戳/，然后覆盖为新配置

▲ 配置同步：预览变更 → 确认同步 → 自动备份

💡 提示：如果预览显示「本地配置已是最新版本，无需同步」，说明你已经在使用管理员的最新配置。

6.5 管理员：把最新配置下发到同事

当你根据校准报告和人工审核，确认了一组配置调整后，可以一键同步给所有同事：

1

8. 常见问题

Q: 启动报错 "ModuleNotFoundError: No module named 'xxx'"

A: 确保已安装全部依赖：pip install -r requirements.txt。如果使用了虚拟环境，请确认已激活。

Q: 评审返回错误 "未配置 API Key"

A: 检查 .env 文件中是否已填入对应 LLM_PROVIDER 的 API Key。API Key 不能是 sk-xxx 或 xxx 这样的占位值。

Q: 可以同时使用多个模型吗？

A: 单次评审使用一个模型，但可以在 Gradio 界面或 API 调用时每次选择不同的模型提供商。配置所有 4 个 API Key 后可以灵活切换。

Q: 评审一篇论文大概多长时间？

A: 取决于论文长度和模型响应速度，通常在 30-120 秒。5 个 Agent 分两轮并行执行，Round 1（A1+A2+A3）和 Round 2（A4+A5）各等待一次 LLM 响应。

Q: 校准引擎需要多少文章才能运行？

A: 最少需要 2 篇获奖 + 2 篇失败 文章。但建议积累至少 10 篇以上再运行，以获得有统计意义的结果。外部获奖文章是可选的。

Q: 数据库和向量数据存储在哪里？

A: SQLite 数据库存储在 data/academic_reviewer.db，ChromaDB 向量数据存储在 data/chroma/。这些都是本地文件，无需额外安装数据库服务。

Q: 如何修改 A4 Agent 的改写边界（Iron Law）？

A: A4 的改写限制在提示词中定义（prompts/a4_language_style.txt）。当前规则：仅拼写/语法/标点/中式英语可以直接改写；风格/词汇/结构调整只能给出建议。如需调整，直接修改提示词文件即可。

Q: 支持哪些文件格式？

A: 支持 .txt、.pdf、.docx 三种格式。PDF 使用 PyPDF2 提取文本，如果 PDF 是扫描版（图片），提取效果可能不理想。建议优先使用 .txt 或 .docx 格式。

Q: 教师经验文档需要 LLM 吗？成本如何？

A: 取决于选择的模式。Mode 3（完全结构化，使用 ### [pattern] 特征名 格式）不需要 LLM，零成本。Mode 2（半结构化，使用系统提供的章节标题分节）和 Mode 1（纯自由格式随笔）需要调一次 LLM 来完成文本到知识点的拆分，成本约等同于一次评审调用的 1/3。对于同一个文档，LLM 只调用一次。在系统未配置 API Key 的情况下，Mode 1 和 Mode 2 会报错并提示需要 LLM。

Q: 同事怎么留下自己的名字？数据如何区分？

A: 同事只需在 .env 中设置 INSTANCE_NAME=刘老师（可以写任意名称）。每次评审或校准自动上报时会带上这个名字，管理员打开「管理看板」就能按名字区分不同同事的数据。如果不填，系统会使用电脑主机名作为标识。

Q: 同事不上报数据也能正常使用系统吗？

A: 可以。如果同事的 .env 中 SYNC_SERVER_URL 为空，系统完全在本地运行，不上报任何数据。团队协作功能是可选、可配置的，不影响评审和校准的全部核心功能。

Q: 管理员调优了评分标准，我怎么同步到最新？

A: 打开系统 → 「系统状态」标签页 → 展开底部「从中央服务器同步最新配置」→ 点击「预览变更」查看变化 → 点击「确认同步」完成覆盖。旧配置会自动备份到 data/configs_backup_时间戳/，不用担心丢失。

💡 使用建议

  • 先跑校准再调配置：积累一定量的评审数据后运行校准引擎，用数据驱动的方式优化评分阈值
  • 多模型对比：同一篇论文用不同模型评审，观察评分一致性
  • 定期查历史：利用评审历史功能追踪学生的进步轨迹
  • A5 原创性检查：将已批改的优秀论文索引到 ChromaDB，后续稿件自动查重
